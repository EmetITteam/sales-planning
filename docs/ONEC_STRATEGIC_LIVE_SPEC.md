# ТЗ для 1С-розробника: `getSalesLineItems` — live-вигрузка продажів для дашборду «Стратегія»

**Дата:** 2026-07-03 · **Замовник:** sales-planning (дашборд `/admin/strategic-kpi`)

## Контекст (навіщо це)

Дашборд «Стратегія» рахує KPI (унікальні клієнти, купують/міс., ср/уп, середній чек, категорії клієнтів, промо) з таблиці `sales` — це **порядкові рядки реалізацій** (line-items), ті самі що у Excel-звіті **«Продажи»**. Історія 2022→червень-2026 (265 тис. рядків) уже завантажена разово з TSV-вигрузки.

Тепер треба **тримати поточний місяць живим**: минулі місяці беремо з нашої БД, а **поточний місяць періодично підтягуємо з 1С** і докладаємо у БД. Один місяць — це всього **~5–6 тис. рядків**, тому вигрузка легка (це НЕ вся історія).

**Що треба від 1С:** один HTTP-екшен, який за діапазон дат повертає рядки звіту «Продажи» у JSON — сирими полями, БЕЗ фільтрації/класифікації (класифікацію по брендах/подарунках робить наша сторона).

---

## Action: `getSalesLineItems`

### Вхід

```json
{
  "action": "getSalesLineItems",
  "payload": {
    "dateFrom": "2026-07-01",
    "dateTo":   "2026-07-31",
    "page": 1,
    "pageSize": 1000,
    "login": "sdu@emet.in.ua"
  }
}
```

- `dateFrom` / `dateTo` — **включно**, за полем «Дата» документа реалізації (той самий період, що вибирають у звіті «Продажи»).
- `page` / `pageSize` — пагінація (див. нижче). Якщо не передані — `page=1`, `pageSize=1000`.
- `login` — для scope-перевірки (director-proxy `sdu@emet.in.ua`, як у `getRegistryPlans`). Повертати ВСІ підрозділи (без фільтра по менеджеру) — це company-wide звіт.

### Вихід

```json
{
  "status": "success",
  "data": {
    "page": 1,
    "pageSize": 1000,
    "totalRows": 5522,
    "hasMore": true,
    "rows": [
      {
        "docNumber": "ЗИН00016140",
        "docLine": 1,
        "date": "2026-07-05T15:18:27",
        "clientCode": "000008266",
        "clientName": "Іванова Олена Петрівна",
        "phone": "+380501234567",
        "product": "HP CELL VITARAN i 2.0 (2*2.5ml)",
        "discount": "Vitaran від 4х уп. -15% (05.26)",
        "division": "Киев",
        "seller": "Некова Катерина",
        "seminar": "",
        "project": "",
        "qty": 3.0,
        "sumUsd": 315.00
      }
    ]
  }
}
```

### Поля рядка (це рівно колонки звіту «Продажи»)

| JSON-поле | Колонка звіту «Продажи» | Тип | Примітка |
|---|---|---|---|
| `docNumber` | номер документа реалізації | string | Напр. `ЗИН00016140`. **Тільки номер**, без «Реализация... от ...». |
| `docLine` | № рядка в документі | int | 1, 2, 3… Стабільний порядок рядків у межах документа. Пара (`docNumber`,`docLine`) = унікальний ключ рядка. |
| `date` | Дата | string ISO | `YYYY-MM-DDTHH:MM:SS`, київський локальний час, **без TZ-суфікса**. |
| `clientCode` | Код (контрагента) | string | З лідируючими нулями (`000008266`) — НЕ обрізати нулі. |
| `clientName` | Контрагент | string | |
| `phone` | Телефон контрагента | string \| null | |
| `product` | Номенклатура | string | Повна назва (по ній визначаємо бренд). |
| `discount` | Повод скидки | string \| null | Напр. `Подарок Vitaran (05.26)`, `Амбассадор`, `День Рождения`. По ньому визначаємо промо/подарунок. |
| `division` | Подразделение | string | Напр. `Киев`, `Одесса`, `Коллцентр Call center лидогенерация`, `Полтава*`. По ньому визначаємо канал. |
| `seller` | Сотрудник | string \| null | |
| `seminar` | Семинар (назва) | string \| null | Для ELLANSE. |
| `seminarDate` | **Дата проведення семінару** | string \| null | `YYYY-MM-DD`. Потрібна щоб рахувати КОЖНУ подію семінару окремо (один семінар у місті може проходити кілька разів різними датами). |
| `project` | Проект (назва) | string \| null | Для ELLANSE. |
| `projectDate` | **Дата проведення проекту** | string \| null | `YYYY-MM-DD`. Аналогічно seminarDate — для проектів. |
| `qty` | Количество (в ед. хранения) | number | Десяткове (3.0). **JSON-число**, не рядок з NBSP. |
| `sumUsd` | Сумма продажи (без НДС) в USD | number | Може бути `0` (подарунковий рядок). **JSON-число**, крапка-decimal. |

**Важливо:**
- Повертати **ВСІ рядки** діапазону — зокрема подарункові (`sumUsd=0`), зі знижками, амбассадорські. Нічого не відфільтровувати — класифікацію (бренд / подарунок / ігнор / промо / канал) робить наша сторона по цих сирих полях.
- `qty` і `sumUsd` — саме **JSON-числа** (`315.00`, не `"315,00"` і не `"1 315,00"` з NBSP).
- `date` — ISO без TZ (`2026-07-05T15:18:27`).

### Пагінація

⚠️ **Вебхук НЕ потрібен.** Пагінація — це просто **кілька окремих POST-запитів З НАШОГО боку**, а не «одна відповідь на кілька сторінок». 1С stateless: на кожен запит повертає свій шматок за `page`/`pageSize`. Один POST → одна відповідь (як завжди). Наступні сторінки **тягнемо ми**, роблячи новий POST:

```
Ми: POST {…, page: 1} → 1С відповідає сторінкою 1
Ми: POST {…, page: 2} → 1С відповідає сторінкою 2
... доки hasMore=false
```

**НАЙПРОСТІШИЙ варіант (рекомендовано для старту):** місяць ≈ 5–6 тис. рядків (~1–2 МБ JSON) — можна віддати **ВЕСЬ діапазон одним POST-ом БЕЗ пагінації**. Тоді `page`/`pageSize` можна не реалізовувати (або ігнорувати), а `hasMore` завжди `false`. Пагінацію додати ПОТІМ, якщо колись діапазон стане завеликим.

Якщо все ж реалізуєте пагінацію:
- Повертати `pageSize` рядків, `page` — 1-based.
- `totalRows` — скільки всього рядків у діапазоні; `hasMore=true`, якщо є наступна сторінка.
- Порядок рядків **стабільний** між сторінками (сорт по `docNumber`, `docLine`) — інакше сторінки дублюватимуть/губитимуть рядки.

### Порожній діапазон

```json
{ "status": "success", "data": { "page": 1, "pageSize": 1000, "totalRows": 0, "hasMore": false, "rows": [] } }
```

### Помилка

```json
{ "status": "error", "message": "текст помилки" }
```

## Оптимізація (критично — без цього довго)

1. **Ми НЕ тягнемо історію** — тільки поточний (відкритий) місяць. Це ~5–6 тис. рядків, 1–2 МБ JSON. Минулі місяці вже у нашій БД і більше не запитуються.
2. **Full-month replace на нашому боці.** Кожен синк ми забираємо весь поточний місяць і повністю оновлюємо його у себе (старі рядки місяця, яких уже нема у вигрузці — напр. скасовані документи — самі зникають). Тому:
   - **1С НЕ треба вести окремий «фід видалень»** — просто віддавайте актуальний стан діапазону.
   - Скасований/видалений документ = його рядків просто немає у відповіді → у нас вони приберуться.
3. **(Опційно, на майбутнє) `changedSince`.** Якщо колись місяць стане великим — додамо параметр `changedSince: "2026-07-05T12:00:00"` і 1С повертатиме тільки документи, змінені після цієї мітки. Поки НЕ обов'язково (повний місяць і так легкий). Якщо реалізуєте одразу — добре, але не блокуюче.

### Цільова швидкість

- Одна сторінка (1000 рядків) — бажано **< 3–5 сек**.
- Весь місяць (5–6 сторінок) — сумарно **< 20–30 сек**. Ми викликаємо це у фоні (не в момент відкриття дашборду користувачем), тому жорсткого real-time-ліміту немає, але чим швидше — тим свіжіші дані.

## Автентифікація / доступ

- Той самий механізм, що інші company-wide екшени: Basic Auth + `login` у payload = `sdu@emet.in.ua` (director-proxy). Дашборд admin-only, тому scope безпечний.
- `includeAll`-семантика: повертати всі підрозділи (представництва + колл-центр + дистриб'ютори Полтава*/Черновцы* + інші), без фільтра по закріпленню менеджера.

## Що НЕ треба робити

- НЕ класифікувати бренди/подарунки/канали — це наша логіка (може мінятись без 1С).
- НЕ віддавати агреговані суми — потрібні **порядкові рядки** (line-items), бо ми рахуємо унікальних клієнтів, категорії (нові/сплячі/…), промо — це неможливо з агрегатів.
- НЕ фільтрувати нульові/подарункові рядки.

## Приклад узгодження (як перевірити що все ок)

За `dateFrom=2026-06-01, dateTo=2026-06-30` екшен має повернути `totalRows ≈ 5522` (стільки рядків червня у нас зараз у БД з TSV-backfill). Якщо цифра близька — маппінг колонок правильний.
