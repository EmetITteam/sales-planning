# Bitrix SPA 1039 — Верифікація нового клієнта

ТЗ для створення Smart Process у Bitrix24 (формат подібний до SPA 1038 «Рекламації»).

## 1. Базові налаштування SPA

| Параметр | Значення |
|----------|----------|
| Назва | Верифікація нового клієнта |
| Назва множинна | Верифікації клієнтів |
| Назва елемента | Заявка на верифікацію |
| Технічний тип | `DT1039` (або наступний вільний) |
| Шлюз API | Включити (через REST API) |

## 2. Поля SPA

| Назва поля | Тип | Обов'язкове | Технічна назва (підказка) |
|-----------|-----|-------------|---------------------------|
| ПІБ клієнта | Рядок | ✅ | `ufCrm_<...>_CLIENT_NAME` |
| Телефон | Рядок | ✅ | `ufCrm_<...>_CLIENT_PHONE` |
| Адреса/місто | Рядок | ✅ | `ufCrm_<...>_CLIENT_ADDRESS` |
| Тип закладу | Рядок (або довідник) | ❌ | `ufCrm_<...>_INSTITUTION_TYPE` |
| ID клієнта у 1С | Рядок | ✅ | `ufCrm_<...>_CLIENT_ID_1C` |
| Логін менеджера-ініціатора | Рядок (email) | ✅ | `ufCrm_<...>_MANAGER_LOGIN` |
| ПІБ менеджера | Рядок | ❌ | `ufCrm_<...>_MANAGER_NAME` |
| Документи | Файли (множина) | ❌ | `ufCrm_<...>_DOCUMENTS` |
| Освіта клієнта | Рядок | ❌ | `ufCrm_<...>_EDUCATION` |
| Коментар менеджера | Текст | ❌ | `ufCrm_<...>_INITIAL_COMMENT` |

> **⚠️ Після створення передай мені точні технічні назви всіх полів (із префіксом `ufCrm_1039_` чи яким вийде).**

## 3. Стадії (статуси)

Стандартний воронкоподібний flow:

| Stage ID | Назва | Колір | Призначення |
|----------|-------|-------|-------------|
| `NEW` | Створено | сірий | Тільки що подано, чекає на КЦ |
| `IN_PROGRESS` | У роботі | жовтий | КЦ взяв у обробку |
| `CLARIFICATION` | На уточненні | помаранчевий | Чекає коментаря менеджера-ініціатора |
| `DONE` | Верифіковано ✅ | зелений | success — менеджер отримає нотифікацію |
| `REJECTED` | Відхилено ❌ | червоний | failed — менеджер отримає нотифікацію з причиною |

> **⚠️ Передай мені повні Stage ID у форматі `DT1039_<тип>:<stage>` (наприклад `DT1039_4:NEW`, `DT1039_4:DONE`).**

## 4. Permission groups

Той самий принцип як для SPA 1038 (рекламації):

| Роль | Доступ |
|------|--------|
| КЦ-менеджери | Повний (CRUD + зміна статусу + коментарі) |
| Інші (sales / медичний відділ) | Read-only або None |

## 5. Webhook на події

Налаштування у Bitrix → Розробникам → Інші → Вихідний webhook:

| Подія | URL | Призначення |
|-------|-----|-------------|
| `ONCRMDYNAMICITEMUPDATE` для DT1039 | той самий що рекламації (`https://reclamation-app...../api/webhook/bitrix_event`) | Status change → колокольчик у sales-planning |
| `ONCRMTIMELINECOMMENTADD` (опційно) | те саме | Якщо КЦ написав коментар у режимі «На уточненні» — менеджер отримає сповіщення |

Authentication token — той самий що для SPA 1038.

## 6. Коли все готове — передай мені

1. **ID SPA** (entityTypeId) — напр. `1039`
2. **Технічні назви полів** (з префіксом `ufCrm_...`)
3. **Stage IDs** (повні, з префіксом)
4. **Тестову заявку** — створи 1 тестовий запис вручну у Bitrix → подивимось webhook payload разом

Я підставлю константи у код і запустимо.
