# ТЗ для 1С-розробника: Actions 14 + 15 для модуля семінарів

**Контекст:** sales-planning впроваджує модуль адміністрації семінарів. Потрібно витягувати умови з 1С довідника «Виды обучения» і перевіряти виконання цих умов клієнтом по реалізаціях.

---

## Action 14 — `getSeminarConditions`

### Призначення

Витягти умови участі для конкретного семінару (з вкладки «Условия» довідника «Виды обучения»).

### Вхід

```json
{
  "action": "getSeminarConditions",
  "payload": {
    "trainingId": "4670"
  }
}
```

### Вихід

```json
{
  "status": "success",
  "data": {
    "trainingId": "4670",
    "conditions": [
      {
        "nomenclature": "_ELLANSE",
        "brand": "ELLANSE",
        "minAmount": 640,
        "dateFrom": "2026-05-08",
        "dateTo": "2026-06-29"
      }
    ]
  }
}
```

### Логіка

- Знайти у довіднику «Виды обучения» запис з `Код = trainingId`
- Витягти вкладку «Условия» — кожен рядок повертати як умову
- `nomenclature` — поле «Состав номенклатуры» (наприклад `_ELLANSE`)
- `brand` — можна виводити з номенклатури (опційно)
- `minAmount` — поле «Сумма»
- `dateFrom` / `dateTo` — поля «Дата С» / «Дата по»

### Якщо умов нема (порожня вкладка)

```json
{
  "status": "success",
  "data": {
    "trainingId": "4670",
    "conditions": []
  }
}
```

### Якщо семінар не знайдено

```json
{
  "status": "error",
  "message": "Семинар по коду не найден"
}
```

## Action 15 — `checkClientSeminarPurchases` ⭐ КРИТИЧНО

### Призначення

Перевірити чи клієнт виконав умови закупок для семінару — з урахуванням того, що **закупки мають бути на семінарських реалізаціях** (з тегом мероприятия), а не просто загальною сумою по бренду.

### Вхід

```json
{
  "action": "checkClientSeminarPurchases",
  "payload": {
    "clientId": "000023418",
    "trainingId": "4670",
    "login": "sm.kiev1@emet.in.ua"
  }
}
```

`login` — потрібен для scope-check (як у інших actions, IDOR fix).

### Вихід (всі умови виконані)

```json
{
  "status": "success",
  "data": {
    "clientId": "000023418",
    "trainingId": "4670",
    "checkedAt": "2026-06-12T15:30:00Z",
    "conditions": [
      {
        "nomenclature": "_ELLANSE",
        "required": 640,
        "actualSeminarRelevant": 720,
        "passed": true,
        "invoices": [
          {
            "invoiceId": "00012345",
            "date": "2026-06-15",
            "amount": 360,
            "nomenclature": "_ELLANSE",
            "hasSeminarTag": true,
            "linkedTrainingId": "4670"
          },
          {
            "invoiceId": "00012678",
            "date": "2026-06-20",
            "amount": 360,
            "nomenclature": "_ELLANSE",
            "hasSeminarTag": true,
            "linkedTrainingId": "4670"
          }
        ]
      }
    ],
    "allConditionsMet": true
  }
}
```

### Вихід (НЕ виконав)

```json
{
  "status": "success",
  "data": {
    "clientId": "000023418",
    "trainingId": "4670",
    "checkedAt": "2026-06-12T15:30:00Z",
    "conditions": [
      {
        "nomenclature": "_ELLANSE",
        "required": 640,
        "actualSeminarRelevant": 360,
        "passed": false,
        "shortfall": 280,
        "invoices": [
          {
            "invoiceId": "00012345",
            "date": "2026-06-15",
            "amount": 360,
            "nomenclature": "_ELLANSE",
            "hasSeminarTag": true,
            "linkedTrainingId": "4670"
          }
        ]
      }
    ],
    "allConditionsMet": false
  }
}
```

### Логіка

1. **Отримати умови** для `trainingId` (як Action 14)
2. Для кожної умови:
   - Знайти **реалізації** клієнта `clientId` за період `[dateFrom, dateTo]`
   - **Фільтрувати тільки ті, що мають тег мероприятия** = поточний семінар (або взагалі семінарські — узгодити з бізнесом)
   - Фільтрувати по `nomenclature` (співпадає з умовою)
   - Сумувати `amount`
   - Порівняти з `required`
   - `passed = (actualSeminarRelevant >= required)`
3. `allConditionsMet = всі conditions passed=true`
4. У `invoices` повертати деталі для дебагу і прозорості

### Чому це не просто `getSalesFact`

`getSalesFact` повертає сумарну закупку клієнта по сегменту/бренду за період. Це **недостатньо** — нам треба:
- Фільтр по конкретній номенклатурі (не сегменту)
- Фільтр по семінарських реалізаціях (з тегом мероприятия)
- Деталі по конкретним реалізаціях (id, дата, сума)

Це специфічна business-логіка яку розуміє лише 1С.

### Помилки

```json
// Клієнт не належить менеджеру (scope-check, як для інших actions)
{ "status": "error", "code": 403, "message": "Client outside your scope" }

// Клієнт або семінар не знайдено
{ "status": "error", "message": "Клиент не найден" }
{ "status": "error", "message": "Семинар не найден" }
```

## Як ми будемо це використовувати у sales-planning

### При записі клієнта на семінар

```
1. Менеджер обирає клієнта у dialog
2. sales-planning викликає Action 15 (checkClientSeminarPurchases)
3. У dialog показуємо:
   ✅ Умови виконані — закупки $720 з $640 (PASSED)
   ⚠️ Умови НЕ виконані — закупки $360 з $640 (не вистачає $280)
4. Менеджер бачить рішення і записує клієнта (або не записує)
```

### При перегляді списку учасників

Черговий менеджер у `/seminars/[id]/manage` бачить колонку «Умови»:
- ✅ — клієнт виконав (auto з 1С)
- ⚠️ — не виконав (auto)
- ✓ ручна — черговий поставив галочку самостійно (без Action 15)

## Питання до уточнення з бізнесом

1. **Що означає «реалізація з тегом мероприятия»?**
   - Це конкретний реквізит у реалізації типу «Мероприятие = trainingId»?
   - Чи це загальний прапор «семінарська реалізація» без прив'язки до конкретного?
   - Це впливає на фільтрацію в Action 15

2. **Якщо клієнт закупив `_ELLANSE` на $720, але без тегу мероприятия — це зараховується?**
   - Очікую: НЕ зараховується (так каже бізнес)
   - Підтвердити з мед-відділом / РМ

3. **Що з закупками за період які НЕ підпадають під умови?**
   - Сходить просто ігнорувати, не показувати у `invoices`
   - Або показувати з прапором `hasSeminarTag: false` щоб менеджер бачив що закупки є, просто не семінарські
