# Спека: Борд «Візити» — план vs факт активностей по менеджерах

**Статус:** 📝 draft, чекає узгодження + «добро» ITD. Створено 2026-07-10.
**Залежності 1С:** [ONEC_ACTION_ACTIVITY_PLAN_SPEC.md](./ONEC_ACTION_ACTIVITY_PLAN_SPEC.md) (план) + новий експорт візитів (факт-історія) — див. §6.

---

## 1. Мета

Окремий борд, де директор / РОП бачить **план vs факт візитів** по менеджерах: скільки візитів заплановано по кожній категорії клієнта і скільки фактично виконано. З розбивкою по регіонах/менеджерах, drill-down до контрагентів і дат, з коментарем менеджера по кожному візиту.

## 2. Доступ (двохрівневий) — узгоджено 2026-07-10

- Окрема сторінка (напр. `/admin/visits` або `/visits`).
- **Рівень 1 — регіональний (авто):** РМ (роль `rm`) бачить борд **тільки по своєму регіону** — без окремого дозволу, за регіоном з сесії 1С. Дані інших регіонів приховані.
- **Рівень 2 — повний борд (по компанії):** гейт через окрему таблицю `visits_permissions` + сторінка `/admin/visits-permissions` (admin вибирає конкретних користувачів, кому показувати всю компанію). Admin — завжди повний.
- Тобто: `admin || у visits_permissions` → вся компанія; `role=rm` → свій регіон; інакше — нема доступу.

## 3. Дані

### Факт візитів — вже є в нашій БД ✅
Таблиця `meetings` (не потрібно нове сховище). Перевірено 2026-07-10: **500 done-візитів, у 100% заповнено `client_category`**, є `comment`, `date`, `client_id_1c`, `manager_login`.
- Візит = рядок `meetings` зі `status='done'`.
- Поля для борду: `manager_login`, `client_id_1c`, `client_name`, `client_category`, `date`, `comment`, `purpose`.

### План — з 1С ⏳
`getActivityPlan` / `getActivityPlansBulk` (окрема спека). План на менеджера × категорію: скільки візитів має бути.

## 4. Бізнес-правила підрахунку (КЛЮЧОВЕ)

Для кожного менеджера × категорія клієнта рахуємо **три числа**:

1. **Виконано плану (по УНІКАЛЬНИХ клієнтах)** — скільки **різних** контрагентів цієї категорії отримали ≥1 виконаний візит у періоді.
   - 10 візитів до одного активного клієнта = **1** закритий план-клієнт.
   - `COUNT(DISTINCT client_id_1c) WHERE category=X AND status=done AND date in period`.
   - Саме це порівнюємо з планом → **% виконання плану**.
2. **Всього візитів** — сирий `COUNT(*)` візитів (10 візитів = 10). Довідково, поряд з унікальними.
3. **План** — `planMeetings` з 1С для цієї категорії.

**% виконання = унікальні_клієнти / план × 100.** (НЕ всього_візитів / план.)

Агрегація вгору: менеджер → регіон → компанія (сума планів, сума унікальних, сума візитів; % рахуємо з сум).

## 5. UI борду

- **Хедер-картки:** всього план / виконано (унікальні) / % / всього візитів — по всій компанії.
- **Розбивка регіон → менеджер** (акордеон, як на планинг-дашборді): по кожному менеджеру таблиця категорій:

  | Категорія | План | Виконано (унік. клієнтів) | % | Всього візитів |
  |---|---|---|---|---|
  | Активні | 20 | 14 | 70% | 31 |
  | Сплячі | 10 | 6 | 60% | 8 |

- **Drill-down (розгортання):** список **контрагентів + дат** візитів менеджера — рядок на візит: клієнт, категорія, дата, **коментар менеджера**. Розгортається/згортається.
- Кольорове кодування % (good/warn/bad) — як у решти бордів.

## 6. Пайплайн даних (історія + онлайн)

Узгоджено 2026-07-10: **джерело — лише 1С-метод** (файл-вигрузки не буде). Візити відмічаються в НАШОМУ застосунку і пушаться в 1С, тож поточний факт живий у `meetings`; метод 1С потрібен для **історії + звірки**.

| Період | Джерело | Механізм |
|---|---|---|
| **Історія (цей рік, минулі місяці)** | `getVisitsBulk` (1С) | одноразовий backfill → `meetings` |
| **Закриті місяці далі** | `getVisitsBulk` (1С) | щомісячно доливаємо |
| **Поточний місяць** | наша `meetings` (live) + звірка з `getVisitsBulk` | факт вже у нас; періодично звіряємо що в 1С нічого не змінилось |

**⚠️ Потрібен 1С-метод `getVisitsBulk`** — спека: [ONEC_ACTION_VISITS_SPEC.md](./ONEC_ACTION_VISITS_SPEC.md). Bulk по всіх менеджерах за діапазон: `docId`, дата, менеджер(логін+ПІБ), регіон, контрагент(код+назва), **категорія на місяць візиту** (з регістру категорій), коментар, статус. Файл-вигрузки не робимо.

## 7. Наша сторона — обсяг робіт

1. **Доступи:** міграція `visits_permissions` + `/admin/visits-permissions` + гард на сторінці.
2. **Backfill-скрипт** візитів у `meetings` (idempotent по `legacy_1c_id`/`doc_id`), з мапінгом категорій RU→UA.
3. **Агрегація** (RPC або API): менеджер × категорія → {план, унік_клієнти, всього_візитів}; згортка до регіону/компанії.
4. **API** `/api/analytics/visits` — план (getActivityPlansBulk) + факт (meetings) змерджені.
5. **Борд UI** `/admin/visits` — хедер, акордеон регіон→менеджер, drill-down візитів з коментарями.
6. **Live-режим** поточного місяця — виклик 1С-методу візитів (якщо частина візитів ведеться прямо в 1С, а не в застосунку).

## 8. Рішення (узгоджено 2026-07-10)

1. ✅ Історія — **лише 1С-метод** `getVisitsBulk`, файл-вигрузки не буде.
2. ✅ Онлайн — візити ведуться **у нас**, факт live у `meetings`; 1С-метод для історії + звірки (чи не змінилось у 1С).
3. ✅ Доступ — **двохрівневий**: РМ авто по своєму регіону + повний борд по `visits_permissions` (§2).
4. ✅ Категорія — **на місяць візиту**, з регістру категорій 1С (для історії — historical, не поточна).
5. ✅ План — по **унікальних клієнтах** категорії (виконано = клієнт відвіданий ≥1 раз); всього візитів рахуємо окремо.

### Лишилось підтвердити 1С-розробнику
- `getVisitsBulk` може віддати **категорію на місяць візиту** з регістру категорій (для минулих місяців — historical)?
- Обсяг за рік по всіх менеджерах в одному виклику — прийнятно по швидкодії (можливо, чанкувати по місяцях)?
