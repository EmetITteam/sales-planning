# 📡 Спека: розширення Action 5 — параметр `includeAll`

**Для:** Андрія (1С розробник)
**Версія:** v2.7
**Дата:** 2026-05-25

## Контекст

Робимо адмін-вкладку «Огляд компанії» на веб-дашборді. Там показуємо план/факт по **всіх** підрозділах включно з:
- Колл-центр
- Адасса
- Полтава\* (Чугуй)
- Чернівці\* (Хайленко)

Зараз `Action 5 getRegionData` повертає тільки 9 з 13 підрозділів — фільтрує по менеджерах. Підрозділи без менеджерів у певному фільтрі не потрапляють у відповідь.

## Що треба

Додати у `Action 5 getRegionData` опціональний параметр `includeAll: boolean` у `payload`.

### Поведінка

| `includeAll` | Поведінка |
|---|---|
| `false` (default, відсутнє) | **Як зараз** — фільтр по менеджерах, повертає тільки релевантні підрозділи |
| `true` | **Нова** — повертає ВСІ підрозділи компанії з фактом, ігноруючи фільтр |

### Сумісність

Існуючі дашборди (РМ, Director) не передають `includeAll` → нічого не змінюється для них. Це повністю backward-compatible.

## Приклад request

```json
{
  "action": "getRegionData",
  "payload": {
    "login": "sdu@emet.in.ua",
    "period": "2026-05",
    "includeAll": true
  }
}
```

## Приклад response (з `includeAll: true`)

Та сама структура що зараз, просто більше елементів у `regions[]`:

```json

  "status": "success",
  "data": {
    "asOfDate": "2026-05-25",
    "prevMonthAsOfDate": "2026-04-30",

        "regionCode": "KYV",

      },
      /* ... інші 8 представництв (як зараз) ... */

      /* НОВЕ — з'являються нові 4 підрозділи: */
```

## Edge cases

1. **Підрозділ без жодного менеджера у системі**
   - Бажано: повертати з синтетичним менеджером, наприклад:
     ```json
     {
       "managerName": "(без менеджера)",
       "managerLogin": "_division_000000060",
       "segments": [ /* агреговані плани/факт підрозділу */ ],
       "totalPlan": ...,
       "totalFact": ...,
       "totalPrevMonthFact": ...
     }
     ```
   - Альтернатива: порожній `managers: []` — тоді наш adapter покаже підрозділ без деталей.

2. **prevMonthFact для нових підрозділів**
   - Якщо є дані минулого місяця — повертати
   - Якщо нема — `0`

3. **Безпека**
   - Параметр `includeAll` доступний усім — НЕ перевіряйте права у 1С
   - У нашому Next.js backend є гард: тільки `role === 'admin'` може передавати `includeAll: true`. Ви на 1С не виконуєте перевірок.

```bash
  -H "Content-Type: application/json" \
  -d '{
    "action": "getRegionData",
    "payload": {
      "login": "sdu@emet.in.ua",

      "includeAll": true

  }'
```

Очікую — `regions[]` з 13 елементів (зараз 11).

## Як перевірити після твого деплою з нашого боку

```bash
node scripts/diag-divisions.mjs
```

У секції `Action 5: getRegionData` має бути 13 регіонів. Зараз — 11.

## Терміни

Без жорсткого дедлайну. Поки не зроблено — на наша адмін-сторінці показуємо «н/д» для 4-х підрозділів (це ОК).

## Що зміниться на нашій стороні після твого деплою

Нічого — у нашому коді ми **вже** передаємо `includeAll: true` (закомічено `0d3df79`+). Як тільки 1С почне її обробляти — у дашборді автоматично з'являться дані без жодних змін у Next.js коді.
