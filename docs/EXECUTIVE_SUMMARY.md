# Sales-Planning → EMET CRM
## Опис проекту розширення для керівництва

**Версія:** 4.0 · **Дата:** 2026-06-02
**Підготувала:** IT Director EMET
**Призначення:** Інформаційний опис проекту для розуміння обсягу робіт, технологій, термінів і витрат

---

## 1. Суть проекту

Будуємо **повноцінну CRM-систему EMET для відділу продажу** — на основі вже працюючого у проді з травня 2026 продукту sales-planning. Не починаємо з нуля: ми **беремо існуючий продукт як базу** і додаємо нові функціональні блоки.

**Ключова ціль:** **вивести щоденну роботу менеджера з продажу з інтерфейсу 1С** у зручний веб- і мобільний сервіс. Менеджер припиняє користуватись 1С як основним інструментом — переходить у нашу CRM, де **все в одному місці**:

- зустрічі з клієнтами фіксуються разом з геолокацією прямо з телефону у полі;
- замовлення оформлюються у швидкій формі з кошиком, контрактом, доставкою і подарунками;
- дебіторка по клієнту видно одразу на його картці, а не у окремій 1С-формі;
- **новий блок рекламацій** з'являється у CRM (раніше менеджер вів їх через окремий незручний Telegram-канал — тепер усе структуровано в одному місці з рештою даних клієнта);
- історія продажів і покупок клієнта відкривається у drill-down з нашого UI.

1С залишається **джерелом правди для бізнес-даних**, але менеджер бачить дані через наш інтерфейс і вносить через наш інтерфейс. CRM-надбудова синхронізує все з 1С автоматично (батчами, не дергаючи 1С на кожен клік).

**Орієнтовний обсяг розробки:** ~3 місяці frontend + паралельна черга 1С-розробника.
**Нові щомісячні витрати на інфраструктуру:** **$51/місяць ($612/рік)** — конкретний склад у Секції 7.

## 2. Що вже зроблено (поточний стан)

Sales-planning **у проді з травня 2026** і покриває 3 повноцінні модулі. Це не прототип — це працююча production-система якою користуються менеджери і керівництво щодня.

### 2.1 Працюючі модулі у проді

| Модуль | Що дає менеджеру | Що дає керівництву |
|---|---|---|
| **Планування продажів** | Місячне планування цілей по 9 ТМ з прив'язкою до клієнтів, finalize-механіка, window-lock | Картина заповнюваності планів по регіонах + по менеджерах, контроль qualified-завершень |

| **Огляд компанії** | — | Виконання плану по регіонах, брендах, менеджерах + аналітика «відстають від плану», структура продажів |

### 2.2 Технічний фундамент

Архітектура зріла і готова до розширення:

- **Сучасний стек:** Next.js 16 + React 19 + TypeScript + Tailwind 4 — поточний промислово-стандартний набір
- **13 інтеграцій з 1С** через захищений HTTP proxy + сесії на JWT
- **240 автоматичних тестів** + перевірка архітектурних інваріантів перед кожним пушем у проді
- **Hosted на Vercel** з автоматичним деплоєм при кожному merge у master
- **Mobile-friendly** дизайн (нещодавно довели до 100% адаптивності)
- **Загалом ~57 000 рядків коду нетто**, ~91 година фактичної розробки

## 3. Поточні болі менеджера, які закриваємо

Усе кругом одного — менеджер витрачає час на 1С-інтерфейс замість роботи з клієнтом:

| Біль | Що це означає у щоденній роботі |
|---|---|
| Менеджер працює у трьох системах — 1С (продажі, дебіторка, замовлення), окремий додаток meeting-4.0 (зустрічі), окремий Telegram-канал для рекламацій | Постійне перемикання контексту, забуті дії, помилки переносу даних |
| Інтерфейс 1С повільний і не оптимізований під щоденну роботу — менеджер витрачає клікі на навігацію, фільтри, пошук клієнта | Кілька хвилин на операцію, що мала би бути за 10 секунд |
| Меню-додаток meeting-4.0 застарів — 2783 рядки коду у одному файлі, геолокація з 5 болями (тихо зберігає null, без retry, без manual fallback) | Скарги менеджерів у полі, втрата геолокації візитів |
| Рекламації менеджер веде через окремий Telegram-канал — без структури, без прив'язки до клієнта, без статусів, без історії | Втрачаються повідомлення, важко відстежити стан, немає єдиного джерела по клієнту |
| Дебіторка видно тільки в 1С — менеджер не бачить її на client card | Менеджер не знає що клієнт винен; пропускає важливі дзвінки про оплату |
| Замовлення створюються у 1С-формі без зручного кошика, фільтрів, перевірок наявності | Помилки у складі замовлень, забуті подарунки, неправильні контракти |
| Аналітика продажів по клієнтах доступна тільки агрегатами по бренду — не видно «що клієнт купував детально» | Втрачаються insights, менеджер не готується точково до візиту |

## 4. Що додаємо — 4 нових блоки + 1 паралельний

### Блок 1 · Зустрічі (Meetings) — 3-4 тижні

Переписуємо meeting-4.0 у наш стек з покращеннями. Менеджер після цього блоку перестає користуватись окремим додатком зустрічей:

| Покращення | Що це дає |
|---|---|
| Геолокація — read-only після захоплення, явний toast при помилці, manual fallback, конфігурований timeout | Менеджер у полі точно знає, що з геолокацією; зникає прихований null-save |
| Buffer pattern — UI не чекає 1С, дані зберігаються миттєво у нашу БД, потім батчем у 1С | Швидкість UI; робота не зупиняється при тимчасових проблемах з 1С |
| Sync-status badge на кожному записі (pending / synced / failed) | Менеджер бачить що з його даними |
| Окремий worker Google Calendar з retry і dead-letter queue | Calendar події не зникають мовчки якщо Google API впав |
| Mobile-first дизайн з Bottom-sheet формою на телефоні | Менеджер швидко створює зустріч у полі без переходу на повну сторінку |

### Блок 1.5 · Деталізація продажів — 1.5 тижні (паралельно з Блоком 1)

**Фундамент для роботи менеджера з історією клієнта на нашій стороні** (без походів у 1С):

- Бекфіл рядкових продажів з 1С у нашу БД (з січня 2025) — товар, кількість, знижка, сума
- Щоночні апдейти + годинні апдейти для «сьогодні»
- Менеджер бачить детальний drill-down «що клієнт купував» прямо на картці клієнта
- On-demand експорт звітів менеджерам без блокування 1С
- Додатково — фундамент для аналітичних віджетів керівництву (Best Manager, top-products, тренди)

### Блок 2A · Дебіторка — 1.5 тижні

- Список дебіторів менеджера з aging (30/60/90/90+ днів)
- Widget «Заборгованість X, протермінована Y» на client card
- Деталь по клієнту: документи, проводки

### Блок 2B · Рекламації — 1 тиждень (новий блок у CRM)

**Перший раз менеджер працює з рекламаціями у CRM, а не у окремому Telegram-каналі.**

Раніше рекламації йшли у незручний загальний Telegram-канал без структури, без прив'язки до клієнта, без статусів і без історії. Тепер:

- Існуюча Bitrix24 + Telegram-інфраструктура зберігається (не ламаємо що працює — там тримається процес обробки)
- У нашій CRM додаємо новий блок: на картці клієнта з'являється секція рекламацій
- Widget-лічильник активних рекламацій + клік відкриває структуровану історію
- Усе в одному екрані з рештою інформації про клієнта (продажі, дебіторка, зустрічі)
- Менеджер припиняє стежити за Telegram-каналом — він бачить свої активні рекламації одразу на дашборді

### Блок 3 · Замовлення / Реалізації — 3-4 тижні

**Менеджер припиняє оформлювати замовлення у 1С.** Базуємось на UI-каркасі з meeting-app (WIP-версія з заказами):

- Швидка форма зі зручним кошиком, пошуком товарів, вибором контракту
- Multi-currency UAH/USD з автоперерахунком при зміні контракту
- Gift logic (вибір подарунків, перевірка обмежень, tracking)
- Status workflow: чернетка → проведено → реалізація
- 9 нових 1С actions для повної інтеграції

## 5. Технологічний підхід

### Принципи

| Принцип | Що означає |
|---|---|
| **1С = source of truth** | Усі бізнес-дані остаточно живуть в 1С. Ми надбудова. Не дублюємо бізнес-логіку. |
| **Швидкість UI важливіша за realtime sync** | Менеджер не чекає 1С на кожен клік — данні пишуться у нашу БД миттєво, синхронізуються батчем |
| **Mobile-first для CRM** | Менеджери у полі. Все що ми робимо — спочатку працює на телефоні |
| **Безпека за замовчуванням** | RLS + JWT + закриття IDOR перед першим фінансово-чутливим блоком |
| **Не переробляємо чуже на новий стек** | Reclamations backend лишається Python+Bitrix. Ми додаємо інтеграцію у CRM, не дублюємо систему |

### Стек

- **Frontend:** Next.js 16, React 19, TypeScript, Tailwind 4, lucide-react icons, Plus Jakarta Sans + JetBrains Mono шрифти
- **База даних:** Supabase Postgres з RLS-політиками, перехід на Pro tier перед фінансовими даними
- **Хостинг:** Vercel (existing) + Vercel Cron для регулярних завдань синхронізації
- **Інтеграції:** 1С HTTP-service (existing 13 + 16 нових actions), Google Calendar API, Bitrix24 REST, Telegram Bot API
- **Тестування:** 240 одиничних тестів (існуючий baseline) + Playwright для критичних мобільних сценаріїв
- **Моніторинг:** Sentry для error tracking + session replay (підключаємо паралельно зі стартом)
- **Безпека:** JWT cookies, RLS, secret rotation, ECDSA підписи

## 8. Очікувана бізнес-цінність

| Outcome | Терміни появи |
|---|---|

| Менеджер бачить детальну історію продажів клієнта без походу у 1С | Після Блоку 1.5 (~5-6 тижнів) |
| Аналітичні віджети для керівництва (Best Manager + інші) на основі даних що ми вже маємо | Після Блоку 1.5 (~5-6 тижнів) |
| Дебіторка на client card — менеджер бачить заборгованість одразу при відкритті клієнта | Після Блоку 2A (~7-8 тижнів) |
| Рекламації переходять з Telegram-каналу у структуровану секцію CRM | Після Блоку 2B (~8 тижнів) |
| Замовлення створюються у нашому UI замість 1С — швидко, з кошиком, перевірками, подарунками | Після Блоку 3 (~12-13 тижнів) |

### Якісні вигоди

- Менеджери перестають перемикатись між кількома системами і повільним 1С-інтерфейсом
- Менеджер фокусується на роботі з клієнтом, не на навігації між формами 1С
- Дані фінансово-чутливих модулів захищені RLS і резервуються через PITR
- Керівництво отримує живі дашборди замість Excel-вивантажень
- Помилки у проді знаходимо за хвилини через Sentry session replay

## 9. Ризики і як ми їх мінімізуємо

| Ризик | Імовірність | Вплив | Як мінімізуємо |
|---|---|---|---|
| 1С-черга на нові actions затримує етапи | Висока | Високий | Готуємо повну спеку одразу на ВСІ 16 нових actions, 1С-розробник стартує паралельно. Якщо затримка — frontend все одно готовий і чекає |

| Менеджери чинять опір новому UI | Низька | Середній | Зберігаємо існуючі workflow і термінологію; canary-rollout на 1-2 менеджерах перед mass deploy |

| 1С не повертає `doc_seminar` | Середня | Високий | Без цього у аналітичні віджети попадуть закупки спікерів. Узгоджуємо при спеці; запасний — окремий список лекторів |
