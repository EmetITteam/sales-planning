# Sales-Planning → EMET CRM
## Опис проекту розширення для керівництва

**Версія:** 2.0 · **Дата:** 2026-06-02
**Підготувала:** IT Director EMET спільно з Claude (AI-консультант)
**Призначення:** Інформаційний опис проекту для розуміння обсягу робіт, технологій, термінів і витрат
**Технічний детальний план:** `docs/PROJECT_PLAN.md` (1100+ рядків — для розробки і 1С-команди)

## 1. У двох словах

Перетворюємо поточну систему `sales-planning` (3 модулі у проді з травня 2026) у **повноцінну CRM EMET** для відділу продажу. Виносимо у зручний веб- і мобільний інтерфейс щоденну роботу менеджерів: зустрічі, заказы, дебіторку, рекламації — і автоматизуємо критичну аналітику для керівництва, включно з конкурсом «Найкращий менеджер по ТМ».

**Орієнтовний обсяг:** ~3 місяці frontend-розробки + паралельна черга 1С-розробника.
**Нові щомісячні витрати на інфраструктуру:** ~$25-71/міс (залежно від набору сервісів).

---

## 2. Що вже працює (поточний стан)

Sales-planning у проді з травня 2026, покриває 3 модулі:

- **Модуль планування** — менеджери планують місячні цілі по 9 ТМ, директор бачить заповнюваність
- **Мої клієнти** — менеджер бачить свою клієнтську базу з фільтрами і drill-down
- **Огляд компанії** — директор бачить виконання плану по регіонах і брендах

**Технічний фундамент уже зрілий:**

- Сучасний стек: Next.js 16 + React 19 + TypeScript + Tailwind
- 13 інтеграцій з 1С через захищений proxy + JWT-сесії
- 240 автоматичних тестів + перевірка архітектурних інваріантів перед кожним пушем
- Hosted на Vercel з auto-deploy
- Загалом ~57k рядків коду нетто, ~91 година фактичної розробки

Цей фундамент дозволяє **не починати з нуля** — будуємо CRM поверх існуючого, переписуючи лише meeting-функціонал.

## 3. Поточні болі, які закриваємо

| Проблема | Біль |
|---|---|
| Менеджер працює у трьох системах — 1С (продажі, дебіторка) + meeting-4.0 (зустрічі) + Bitrix24 (рекламації) | Втрата часу на перемикання, забуті обов'язки, помилки переносу даних |
| Додаток meeting-4.0 застарів — 2783 рядки коду у одному файлі, геолокація з 5 болями (тихо зберігає null, без retry, без manual fallback) | Скарги менеджерів у полі, втрата геолокації візитів |
| Конкурс «Найкращий менеджер» рахується вручну у Excel — без виключення спікерів автоматично, без живого drill-down | Час керівництва на рутинні підрахунки, помилки, затримка результатів |
| Дебіторка видно тільки в 1С — менеджер не бачить її на client card | Менеджер не знає що клієнт винен; пропускає важливі дзвінки про оплату |
| Аналітика продажів по клієнтах доступна тільки агрегатами по бренду | Не можна побачити «що клієнт купував детально», втрачаються insights |

## 4. Що додаємо — 4 етапи + 1 паралельний

### Етап 1 · Зустрічі (Meetings) — 3-4 тижні

Переписуємо meeting-4.0 у наш стек з покращеннями:

| Покращення | Що це дає |
|---|---|
| Геолокація — read-only після захоплення, явний toast при помилці, manual fallback, конфігурований timeout | Менеджер у полі точно знає, що з геолокацією; зникає прихований null-save |
| Buffer pattern — UI не чекає 1С, дані зберігаються миттєво у нашу БД, потім батчем у 1С | Швидкість UI; робота не зупиняється при тимчасових проблемах з 1С |
| Sync-status badge на кожному записі (pending / synced / failed) | Менеджер бачить що з його даними |
| Окремий worker Google Calendar з retry і dead-letter queue | Calendar події не зникають мовчки якщо Google API впав |
| Mobile-first дизайн з Bottom-sheet формою на телефоні | Менеджер швидко створює зустріч у полі без переходу на повну сторінку |

### Етап 1.5 · Sales Detail + Best Manager — 1.5 тижні (паралельно зі Stage 1)

**Фундамент для аналітики:**

- Бекфіл деталізованих продажів з 1С у нашу БД (з січня 2025)
- Щоночні апдейти + годинні апдейти для «сьогодні»
- Дозволяє drill-down «що клієнт купував детально», on-demand експорт звітів, тренди

**🏆 Best Manager analytics widget** (на Огляді компанії + Дашборді директора):

Автоматичний розрахунок переможця конкурсу «Найкращий менеджер по ТМ» за 5 брендами (Ellanse / PETARAN / ESSE / IUSE / Vitaran) за прийнятими умовами червня 2026:
- Враховуються пороги участі (наприклад, ≥20 упаковок Ellanse, ≥$4,000 ESSE)
- Універсальні фільтри: індивідуальний план ≥ 100%, план по ТМ ≥ 100%
- Автоматичне виключення закупок спікерів (з документу «Семінар»)
- Тайбрейкер: найвищий % виконання плану по ТМ
- Drill-down: всі учасники рейтингу з прозорими цифрами

### Етап 2A · Дебіторка — 1.5 тижні

- Список дебіторів менеджера з aging (30/60/90/90+ днів)
- Widget «Заборгованість X, протермінована Y» на client card
- Деталь по клієнту: документи, проводки

### Етап 2B · Рекламації — 1 тиждень

- Інтегруємо існуючий reclamation-app (Python + Bitrix24 + Telegram-бот)
- Widget-лічильник активних рекламацій на client card
- Клік → відкриває reclamation-app у новій вкладці
- Зберігаємо існуючу систему, додаємо тільки тонкий міст

### Етап 3 · Заказы / Реалізації — 3-4 тижні

- Базується на UI-каркасі з meeting-app (твоя WIP версія з заказами)
- Multi-currency UAH/USD з автоперерахунком при зміні контракту
- Gift logic (вибір подарунків, перевірка обмежень, tracking)
- Status workflow: чернетка → проведено → реалізація
- 9 нових 1С actions для повної інтеграції

## 5. Технологічний підхід

### Принципи

| Принцип | Що означає |
|---|---|
| **1С = source of truth** | Усі бізнес-дані остаточно живуть в 1С. Ми надбудова. Не дублюємо бізнес-логіку. |
| **Швидкість UI важливіша за realtime sync** | Менеджер не чекає 1С на кожен клік — данні пишуться у нашу БД миттєво, синхронізуються батчем |
| **Mobile-first для CRM** | Менеджери у полі. Все що ми робимо — спочатку працює на телефоні |
| **Безпека за замовчуванням** | RLS + JWT + закриття IDOR перед першим фінансово-чутливим блоком |
| **Не переробляємо чуже на новий стек** | Reclamations лишаються Python+Bitrix. Ми додаємо інтеграцію, не дублюємо систему |

### Стек

- **Frontend:** Next.js 16, React 19, TypeScript, Tailwind 4, lucide-react icons, Plus Jakarta Sans + JetBrains Mono шрифти
- **База даних:** Supabase Postgres з RLS-політиками, перехід на Pro tier перед фінансовими даними
- **Хостинг:** Vercel (existing) + Vercel Cron для регулярних завдань
- **Інтеграції:** 1С HTTP-service (existing 13 + 17 нових actions), Google Calendar API, Bitrix24 REST, Telegram Bot API
- **Тестування:** 240 одиничних тестів (існуючий baseline) + Playwright для критичних мобільних сценаріїв
- **Безпека:** JWT cookies, RLS, secret rotation, ECDSA підписи

| Outcome | Терміни появи |
|---|---|
| Менеджери відмовляються від meeting-4.0, переходять на наш модуль зустрічей | Після Stage 1 (~3-4 тижні) |
| Автоматичний Best Manager widget — керівництво бачить переможців конкурсу без Excel і ручних обчислень | Після Stage 1.5 (~5-6 тижнів) |
| Дебіторка на client card — менеджер бачить заборгованість одразу при відкритті клієнта | Після Stage 2A (~7-8 тижнів) |
| Один екран для рекламацій — лічильник на client card + швидкий перехід | Після Stage 2B (~8 тижнів) |

### Якісні вигоди

- Менеджери перестають перемикатись між трьома системами
- Керівництво перестає чекати ручних Excel-звітів
- Дані фінансово-чутливих модулів захищені RLS і резервуються через PITR
- Конкурс «Найкращий менеджер» прозорий і живий

## 9. Ризики і як ми їх мінімізуємо

| Ризик | Імовірність | Вплив | Як мінімізуємо |
|---|---|---|---|
| 1С-черга на нові actions затримує етапи | Висока | Високий | Готуємо повну спеку одразу на ВСІ 17 нових actions, 1С-розробник стартує паралельно з нашою роботою. Якщо затримка — frontend все одно готовий і чекає, не тратимо |
| Buffer-pattern conflicts (одночасна правка + sync) | Середня | Високий | Per-record advisory locks у БД + state machine. Тестуємо у Sprint 1.5 |
| Supabase Free ліміти на CRM-обсязі (~95 МБ на бекфіл sales detail з 2025) | Висока | Високий | Перехід на Pro перед Stage 1.5 |
| Google Calendar API rate-limits при більшому об'ємі | Середня | Середній | Окремий worker з batch + backoff (а не fire-and-forget як у meeting-4.0) |
| Геолокація у проблемних зонах | Висока | Середній | Manual fallback + явний toast (а не тихе null-збереження) |
| Менеджери чинять опір новому UI | Низька | Середній | Зберігаємо існуючі workflow і термінологію; canary-rollout на 1-2 менеджерах перед mass deploy |
| 1С не повертає `segment_code` у line-item | Середня | Високий | Без цього Best Manager не точний. Узгоджуємо при спеці; запасний план — мапінг через назву продукта на нашій стороні |
| 1С не повертає `doc_seminar` | Середня | Високий | Без цього у Best Manager попадуть закупки спікерів. Узгоджуємо при спеці; запасний — окремий список лекторів |
