# ТЗ для 1С-розробника: Action `getVisitsBulk` — вигрузка візитів (факт)

**Мета:** віддати одним викликом **усі виконані візити** (зустрічі) по всіх менеджерах за діапазон дат — для наповнення історії борду «Візити» та періодичної звірки. Файл-вигрузки не робимо, тільки метод.

Формат виклику — стандартний (той самий web-сервіс, новий `action`; Basic Auth + `X-Service-Token`):
```json
{ "action": "getVisitsBulk", "payload": { ... } }
{ "status": "success", "data": { ... } }
{ "status": "error", "message": "..." }
```

---

## Контекст (як ми це використаємо)

- Візити **реєструються в нашому застосунку** і пушаться у 1С. Тому для поточного місяця факт у нас уже є.
- Цей метод потрібен, щоб:
  1. **Наповнити історію** (минулі місяці цього року) — одноразово.
  2. **Щомісяця доливати** закритий місяць.
  3. **Звіряти** — чи не змінилось щось на стороні 1С (видалили/відредагували візит).
- Матчимо візити по `docId` (= наш `legacy_1c_id`), тож повторний виклик **ідемпотентний** у нас.

---

## Запит

```json
{
  "action": "getVisitsBulk",
  "payload": {
    "dateFrom": "2026-01-01",
    "dateTo": "2026-01-31"
  }
}
```

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `dateFrom` | string | Так | Початок діапазону `YYYY-MM-DD` (по даті візиту). |
| `dateTo` | string | Так | Кінець діапазону `YYYY-MM-DD` включно. |
| `login` | string | Ні | Якщо передано — тільки візити цього менеджера (для точкової звірки). Без нього — усі менеджери. |

> **Швидкодія:** рік по всіх менеджерах може бути важким. Ми будемо викликати **помісячно** (dateFrom/dateTo = межі місяця) і склеювати. Тобто метод має відпрацьовувати місячний зріз швидко; повний рік одним запитом не обов'язковий.

```json
    "visits": [

        "category": "Активный",

        "status": "done"
      }

  }

```

### Поля `Visit`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `docId` | string | Так | Унікальний ID візиту/документа у 1С. Стабільний (не змінюється при редагуванні) — по ньому матчимо/оновлюємо. |
| `date` | string | Так | Дата візиту `YYYY-MM-DD` (можна з часом `YYYY-MM-DDThh:mm:ss`). |
| `managerLogin` | string | Так | Логін менеджера, що провів візит. |
| `managerName` | string \| null | Ні | ПІБ менеджера (зручно для звіту). |
| `region` | string \| null | Так | Регіон/підрозділ менеджера — для розбивки по регіонах. |
| `clientCode` | string | Так | Код контрагента (текст, з провідними нулями: `000008266`). |
| `clientName` | string | Так | Назва контрагента. |
| `category` | string | Так | **Категорія клієнта на МІСЯЦЬ візиту** — з регістру категорій за той місяць (не поточна!). RU-формулювання: `Активный / Спящий / Потерянный / Без закупок / Новый`. Для минулих місяців — historical-значення. |
| `comment` | string \| null | Ні | Коментар менеджера по візиту. |
| `status` | string | Так | Статус візиту. Повертати **тільки виконані** (`done` / «Проведено»). Якщо у 1С є скасовані/заплановані — не включати (або окреме поле, але нам треба лише виконані). |

## Логіка в 1С

1. Вибрати документи «Візит»/«Зустріч» зі статусом **виконано**, де дата ∈ `[dateFrom; dateTo]`.
2. Якщо переданий `login` — фільтр по менеджеру.
3. Для кожного візиту:
   - взяти менеджера (логін + ПІБ), регіон;
   - контрагент (код + назва);
   - **категорію з регістру категорій за місяць дати візиту** (ключове — historical);
   - коментар, статус.
4. Повернути плоский масив `visits[]`. Порожній `[]` якщо нема — не помилка.

## Edge cases

- Немає візитів у діапазоні → `visits: []` (не помилка).
- Візит без коментаря → `comment: null`.
- Клієнт без категорії в тому місяці → `category` = фактична з регістру або `"Без закупок"` за замовчуванням (узгодити).
- Один візит до кількох контрагентів (малоймовірно) → окремий рядок на контрагента.

## Performance

- Цільова швидкодія місячного зрізу по всіх менеджерах: прийнятно кілька секунд (ми викликаємо у фоні/backfill, не в UI-блокуванні).
- Ми чанкуємо по місяцях самі.

---

## Як Sales Planning використає (для довідки)

- **Backfill-скрипт** (аналог `analytics-sales-backfill.mjs`): цикл по місяцях `getVisitsBulk` → upsert у `meetings` по `docId` (`legacy_1c_id`), мапінг категорій RU→UA.
- **Звірка:** періодично тягнемо поточний місяць і порівнюємо з `meetings` — розбіжності підсвічуємо.
- **Борд** рахує: `COUNT(DISTINCT clientCode)` по менеджеру×категорії = виконано плану; `COUNT(*)` = всього візитів.
