# ТЗ для 1С-розробника: HTTP-сервіси для Sales Planning (Спринт 1)

> **Версія:** 2.3 від 05.05.2026
> **Контекст:** Уніфікований HTTP-сервіс для 3 систем (СРМ Мітинг, Планування, Аналітика).
> Спринт 1 — Планування. Розширюємо існуючий сервіс з Мітингу, нічого не ламаємо.

> ### 🆕 Зміни у v2.3 (05.05.2026, друга редакція)
> На основі другого раунду відгуку директора з продажів — фінальне узгодження логіки prev-month:
> - **Action 5 `getRegionData` — змінено семантику prev-month полів.** Замість «факт минулого місяця на той самий N-й робочий день» тепер **повний факт минулого місяця**. Логіка з N-м робочим днем була складна для розуміння менеджерам; директор з продажів попросила показувати простий «повний минулий місяць vs поточний» зверху на дашборді. Стосується: `prevMonthFactUSD`, `prevMonthPlanUSD`, `prevMonthFactPercent`, `totalPrevMonthFact`. Поле `prevMonthAsOfDate` тепер = останній день минулого місяця (раніше було N-й робочий день).
> - **Action 2 `getClientsForPlanning` — категорії клієнтів виправлено на російські.** Спека помилково вказувала українські варіанти, реально 1С віддає російською. Виправлено: `"Активный"`, `"Спящий"`, `"Потерянный"`, `"Новый"`, `"Без закупок"` (з пробілом).
> - **Action 3 `getSalesFact` — параметр `clientIds` явно опціональний.** При пустому масиві або відсутньому полі повертається тільки `totalFactUSD` і `totalClientCount`, `clients` — пустий масив.
>
> ### Зміни у v2.2 (05.05.2026)
> На основі другого раунду відгуку від директора з продажів + питань 1С-розробника:
> - **Action 3 `getSalesFact`:** додано поле `totalClientCount` у segments — кількість унікальних клієнтів які купили цей сегмент у місяці (потрібно для блоку «Незаплановані клієнти» у формі планування). Просте `COUNT(DISTINCT Контрагент)` поряд з `SUM`.
> - **Action 3 + Action 5 `asOfDate`:** виправлено суперечність у дефолтному значенні. Якщо `asOfDate` не передано → береться **останній день місяця `period`** (а не «поточна дата»). Раніше було двозначно: для запиту по минулому місяцю «поточна дата» лежить за межами періода.
>
> ### Зміни у v2.1 (28.04.2026)
> - **Новий Action 6: `getTrainings`** — отримання обучень із справочника "Виды обучения" для блоку "Закриття розриву".
> - **Action 3 `getSalesFact` розширено:** опціональний параметр `asOfDate` — факт на конкретну дату (для порівняння з минулим місяцем на той же N-й робочий день).
> - **Action 5 `getRegionData` розширено:** опціональний `asOfDate` + у відповідь додано поля `prevMonthFactUSD` і `prevMonthFactPercent` для кожного сегменту (порівняння динаміки з минулим місяцем).
>
> Existing actions 1, 2, 4 — без змін.

## Action 1: Розширення `login`

**Мета:** Додати до існуючої відповіді роль, регіон і список підлеглих.

**Запит (без змін):**
```json
{
  "action": "login",
  "payload": {
    "login": "siryk@emet.com",
    "password": "..."
  }
}
```

**Відповідь ЗАРАЗ (що вже є):**
```json
{
  "status": "success",
  "data": {
    "login": "siryk@emet.com",
    "role": "Менеджер",
    "auth": true
  }
}
```

**Відповідь ПІСЛЯ розширення (додаємо нові поля, старі залишаються):**
```json
{
  "status": "success",
  "data": {
    "login": "siryk@emet.com",
    "role": "Менеджер",
    "auth": true,
    "roleCode": "manager",
    "fullName": "Сірик Людмила Олексіївна",
    "region": "Дніпро",
    "regionCode": "DNP",
    "managedUsers": []
  }
}
```

**Нові поля:**

| Поле | Тип | Опис |
|------|-----|------|
| `roleCode` | string | `"manager"` / `"rm"` / `"director"` — для веб-додатку |
| `fullName` | string | ПІБ повністю |
| `region` | string | Назва підрозділу користувача (реквізит Підрозділ) |
| `regionCode` | string | Скорочений код регіону. ⚠️ **Точний формат — на ваш розсуд (як у вас названі Підрозділи в 1С).** Веб-додаток адаптується. У моках ми використовуємо 3-буквені мнемоніки як заглушки: DNP/KYV/ODS/LVV/KHK/ZPR/VNN — замініть на ваш реальний формат (наприклад: код підрозділу, повну назву, або номер) і повідомте веб-розробника. |
| `managedUsers` | array | Для менеджера: `[]`. Для РМ: логіни менеджерів його підрозділу. Для директора: логіни всіх РМ |

**Логіка визначення ролі:**
- Є роль "РМ" (Регіональний менеджер) → `roleCode: "rm"`
- Є роль "Директор з продажу" → `roleCode: "director"`
- Інакше → `roleCode: "manager"`

**Логіка `managedUsers`:**
- Менеджер → порожній масив `[]`
- РМ → всі користувачі з тим самим Підрозділом (крім самого РМ)
- Директор → всі користувачі з роллю РМ

**ВАЖЛИВО:** Старе поле `role` (текстове "Менеджер") залишається як є. Додаємо нове поле `roleCode` з кодом. СРМ Мітинг використовує `role`, Планування використовуватиме `roleCode`. Ніщо не ламається.

## Action 3: `getSalesFact`

**Мета:** Отримати факт продажів за місяць — загальний підсумок по сегменту + деталізацію по конкретних клієнтах зі списку планування.

**Запит:**
```json
{
  "action": "getSalesFact",
  "payload": {
    "login": "siryk@emet.com",
    "period": "2026-04",
    "clientIds": ["00000012345", "00000054321", "00000067890"],
    "asOfDate": "2026-04-15"
  }
}
```

| Параметр | Тип | Опис |
|----------|-----|------|
| `login` | string | Email менеджера |
| `period` | string | Місяць у форматі YYYY-MM |
| `clientIds` | array | **Опціональний.** Масив кодів контрагентів зі списку планування (до 400 штук). Якщо передано — деталізація `clients[]` фільтрується по них. Якщо пустий масив `[]`, `null` або поле відсутнє — повертати `totalFactUSD` + `totalClientCount`, а `clients` як пустий масив `[]`. |
| `asOfDate` | string | **🆕 Опціональний.** Дата у форматі YYYY-MM-DD. Якщо передано — факт рахується з 1-го числа місяця по `asOfDate` включно. Якщо не передано або null — береться **останній день місяця `period`** (тобто повний місяць). Використовується для зрізу «факт станом на сьогодні» при перегляді поточного місяця. |

**Відповідь:**
```json
{
  "status": "success",
  "data": {
    "segments": [
      {
        "segmentCode": "PETARAN",
        "segmentName": "Petaran",
        "totalFactUSD": 18500.00,
        "totalClientCount": 7,
        "clients": [
          {
            "clientId": "00000012345",
            "clientName": "Главацька Д.С.",
            "factAmountUSD": 1250.00
          },
          {
            "clientId": "00000054321",
            "clientName": "Іванова О.П.",
            "factAmountUSD": 1950.00
          }
        ]
      },
      {
        "segmentCode": "NEURAMIS",
        "segmentName": "Neuramis",
        "totalFactUSD": 7200.00,
        "totalClientCount": 3,
        "clients": [
          {
            "clientId": "00000012345",
            "clientName": "Главацька Д.С.",
            "factAmountUSD": 890.00
          }
        ]
      }
    ]
  }
}
```

**Поля segments:**

| Поле | Тип | Опис |
|------|-----|------|
| `segmentCode` | string | Код сегменту |
| `segmentName` | string | Назва сегменту |
| `totalFactUSD` | number | Загальний факт по сегменту за місяць — ВСІ клієнти менеджера, не тільки ті що в clientIds |
| `totalClientCount` | number | **🆕** Загальна кількість унікальних клієнтів менеджера які купували цей сегмент у цьому місяці. Потрібно для розрахунку «Незапланованих» (тих хто купив, але не був у плані менеджера) — frontend обчислює `Незаплановані count = totalClientCount − clients.length`. |
| `clients` | array | Деталізація тільки по клієнтах з clientIds (тих, хто реально купив у цьому місяці) |

**Поля clients:**

| Поле | Тип | Опис |
|------|-----|------|
| `clientId` | string | Код контрагента |
| `clientName` | string | Назва контрагента (скорочена) |
| `factAmountUSD` | number | Сума продажів цього клієнта по цьому сегменту за місяць (USD) |

**Логіка в 1С:**

1. Обороти регістру `Продажі` WHERE Менеджер = login AND Період з 1-го числа місяця по `asOfDate` (або по останній день місяця, якщо `asOfDate` не передано)
2. `totalFactUSD` = SUM(СуммаUSD) GROUP BY Сегмент номенклатури — по ВСІХ клієнтах менеджера
3. `totalClientCount` = COUNT(DISTINCT Контрагент) GROUP BY Сегмент номенклатури — кількість унікальних покупців у цьому місяці (по ВСІХ клієнтах, не тільки з `clientIds`)
4. `clients[]` = SUM(СуммаUSD) GROUP BY Сегмент номенклатури, Контрагент WHERE Контрагент В (&clientIds)
5. Якщо клієнт з clientIds не купував цей бренд у цьому місяці — НЕ включати його в clients[]
6. Сегменти де totalFactUSD = 0 і clients порожній — НЕ включати в відповідь

**Важливо:**
- `totalFactUSD` і `totalClientCount` рахуються по ВСІХ клієнтах менеджера (може бути 1000+), а `clients[]` фільтрується тільки по переданому масиву `clientIds`
- Це різні числа: totalFactUSD ≥ SUM(clients[].factAmountUSD), і totalClientCount ≥ clients.length
- Суми в доларах — вони є в регістрі
- Якщо кілька документів по одному клієнту/бренду — сумувати в один рядок

**🆕 Use case для `totalClientCount`** (для розуміння — реалізовувати нічого додатково не треба):
У формі планування менеджер планує продажі по 3 сегментах клієнтів: Активні / Нові / Активація. Загалом ~12 клієнтів — ці IDs передаються у `clientIds`. Але реально в місяці може купити 15 клієнтів — 12 запланованих + 3 «спонтанних» («Незаплановані» — ті кому менеджер не планував, але вони все одно прийшли і купили). Frontend має показати в таблиці окремий рядок «Незаплановані: N клієнтів, $X». Без `totalClientCount` ми не знаємо `N` (суму $X можемо обчислити як totalFactUSD − Σ clients[]).

## Action 4: `getRegistryPlans`

**Мета:** Отримати плани продажів за період з деталізацією по підрозділах, менеджерах і сегментах.

**Запит:**
```json
{
  "action": "getRegistryPlans",
  "payload": {
    "dateFrom": "2026-04-01",
    "dateTo": "2026-04-30"
  }
}
```

| Параметр | Тип | Опис |
|----------|-----|------|
| `dateFrom` | string | Початок періоду (YYYY-MM-DD) |
| `dateTo` | string | Кінець періоду (YYYY-MM-DD) |

**Відповідь:**

```json
        "managerLogin": "siryk@emet.com",
        "managerName": "Сірик Людмила",
        "segmentCode": "NEURAMIS",
        "segmentName": "Neuramis",
        "planAmountUSD": 5000.00
      },

        "period": "2026-04-01",

```

**Поля plans:**

| Поле | Тип | Опис |
|------|-----|------|
| `period` | string | Дата періоду плану (YYYY-MM-DD) |
| `divisionCode` | string | Код підрозділу |
| `divisionName` | string | Назва підрозділу |
| `managerLogin` | string | Логін менеджера в ІБ 1С (тo само значення що `login` в Action 1 і `managerLogin` в Action 5). Frontend по цьому полю джойнить плани з фактом. |

| `segmentName` | string | Назва сегменту |

**Логіка в 1С:**

1. Регістр `ПлануванняПродажів` (або документ "Планування продажів") за період dateFrom–dateTo
2. Повертати ВСІ рядки: по всіх підрозділах, менеджерах, сегментах
3. Не фільтрувати по конкретному менеджеру — веб-додаток сам фільтрує на своїй стороні

**Важливо:**
- Метод повертає ВСІ плани за період (не фільтрований по одному менеджеру)
- Плани заводяться раз на місяць — даних небагато (~25 менеджерів × 8 сегментів = ~200 рядків)
- Цей метод використовуватиметься і в Плануванні, і в СРМ дашборді, і пізніше в Аналітиці

## Action 5: `getRegionData`

**Мета:** Агреговані дані план/факт по всіх менеджерах регіону. Для РМ та Директора.

**Запит:**
```json
{
  "action": "getRegionData",
  "payload": {
    "login": "rm.dnipro@emet.com",
    "period": "2026-04",
    "asOfDate": "2026-04-15"
  }
}
```

| Параметр | Тип | Опис |
|----------|-----|------|
| `login` | string | Email РМ або Директора |
| `period` | string | Місяць (YYYY-MM) |
| `asOfDate` | string | **🆕 Опціональний.** Дата у форматі YYYY-MM-DD. Якщо передано — факт поточного місяця рахується по цю дату включно. Якщо не передано — береться **останній день місяця `period`** (повний місяць). **Факт минулого місяця завжди рахується ПОВНИЙ** — від 1-го по останній день минулого місяця, незалежно від `asOfDate`. |

**Відповідь:**
```json
{
  "status": "success",
  "data": {
    "region": "Дніпро",
```

**Нові поля у відповіді:**

| Поле | Тип | Опис |
|------|-----|------|
| `asOfDate` | string | Дата на яку рахувався факт поточного місяця (YYYY-MM-DD) |
| `prevMonthAsOfDate` | string | **v2.3:** Останній день минулого місяця (YYYY-MM-DD). Раніше був N-й робочий день минулого, тепер — повний минулий місяць. |

**Поля managers:**

| Поле | Тип | Опис |
|------|-----|------|
| `managerName` | string | ПІБ менеджера |
| `managerLogin` | string | Email менеджера |
| `segments` | array | План/факт по кожному сегменту |
| `totalPlan` | number | Сума плану по всіх сегментах (USD) |
| `totalFact` | number | Сума факту по всіх сегментах (USD) на `asOfDate` |
| `totalPrevMonthFact` | number | **🆕 v2.3:** Сума ПОВНОГО факту минулого місяця (по всіх сегментах) |

**Поля segments:**

| Поле | Тип | Опис |
|------|-----|------|
| `segmentCode` | string | Код сегменту |
| `segmentName` | string | Назва сегменту |
| `planAmountUSD` | number | План (USD) |
| `factAmountUSD` | number | Факт продажів за місяць (USD) на `asOfDate` |
| `prevMonthFactUSD` | number | **🆕 v2.3:** ПОВНИЙ факт минулого місяця (USD) — від 1-го по останній день минулого |
| `prevMonthPlanUSD` | number | **🆕** План минулого місяця (USD) — для розрахунку % виконання |
| `prevMonthFactPercent` | number | **🆕 v2.3:** % виконання плану минулого місяця за повний місяць. Розрахунок: `prevMonthFactUSD / prevMonthPlanUSD * 100` |

**Логіка в 1С:**

1. За логіном визначити роль і підрозділ (з того ж механізму що в `login`)
2. **РМ** → отримати всіх менеджерів свого підрозділу
3. **Директор** → отримати всіх менеджерів всіх підрозділів
4. Визначити `prevMonthAsOfDate` = останній день минулого місяця (наприклад, для `period="2026-04"` → `"2026-03-31"`)
5. По кожному менеджеру:
   - План поточного місяця: з регістру `ПлануванняПродажів` за `period`, GROUP BY сегмент
   - Факт поточного місяця: з оборотів `Продажі` за період [1-е число `period` ; `asOfDate`], GROUP BY сегмент
   - План минулого місяця: з регістру `ПлануванняПродажів` за минулий місяць, GROUP BY сегмент
   - Факт минулого місяця: з оборотів `Продажі` за **повний минулий місяць** [1-е число минулого ; останній день минулого], GROUP BY сегмент
6. `totalPlan` / `totalFact` / `totalPrevMonthFact` = сума по всіх сегментах менеджера

**Чому повний минулий місяць (а не на N-й робочий день):**
Директор з продажів попросила прибрати порівняння «на той самий N-й робочий день» — менеджерам важко це сприймати. Простіша логіка: у дашборді бачиш «факт цього місяця: $X / минулий місяць: $Y / приріст: ±$Z». Минулий місяць — як остаточний підсумок. Поточний — наростаючий результат.

**Важливо:**
- Факт поточного місяця — з 1-го числа місяця по `asOfDate` включно (або по останній день місяця `period` якщо `asOfDate` не передано)
- Факт минулого місяця — **завжди повний** (не залежить від `asOfDate`)
- Сегменти де всі поля (план, факт, prevMonthFact) = 0 — не включати
- Якщо менеджер не має плану в поточному місяці — не включати його
- Якщо в минулому місяці у менеджера не було ні плану ні факту — `prevMonthFactUSD = 0`, `prevMonthFactPercent = 0`
- Якщо в минулому місяці у менеджера не було плану але був факт — `prevMonthFactPercent` повертати як `null` (або 0 — як зручніше)

## Action 6: `getTrainings` 🆕 (v2.1)

**Мета:** Отримати список майбутніх обучень (семінарів, майстер-класів) для регіону менеджера. Використовується у блоці "Закриття розриву" — менеджер обирає, на яке навчання запросить клієнта що довго не купує.

**Запит:**
```json
{
  "action": "getTrainings",
  "payload": {
    "regionCode": "ODS",
    "dateFrom": "2026-04-28"
  }
}
```

| Параметр | Тип | Опис |
|----------|-----|------|
| `regionCode` | string | Код регіону менеджера (як у відповіді `login` — формат на ваш розсуд, див. примітку до Action 1). Приходить з відповіді `login`. |
| `dateFrom` | string | Дата у форматі YYYY-MM-DD. Повертати тільки навчання з `Дата проведения >= dateFrom`. Зазвичай — поточна дата (день, коли менеджер відкрив форму). |

**Відповідь:**
```json
{
  "status": "success",
  "data": {
    "trainings": [
      {
        "trainingId": "4471",
        "trainingName": "ELLANSE Step 2 \"Правильна колагеностимуляція та волюметрія\"",
        "trainingType": "Семінар",
        "date": "2026-05-15",
        "regionCode": "ODS",
        "regionName": "Одеса",
        "city": "04.Одеса"
      },
      {
        "trainingId": "4472",
        "trainingName": "Petaran. Базовий курс",
        "trainingType": "Майстер-клас",
        "date": "2026-05-22",
        "regionCode": "ODS",
        "regionName": "Одеса",
        "city": "04.Одеса"
      }
    ]
  }
}
```

**Поля trainings:**

| Поле | Тип | Опис |
|------|-----|------|
| `trainingId` | string | Код навчання у 1С (поле `Код` з картки навчання) |
| `trainingName` | string | Найменування навчання (поле `Наименование`) |
| `trainingType` | string | Тип/Вид семінара (поле `Вид семинара`: "Семінар", "Майстер-клас" тощо) |
| `date` | string | Дата проведення (поле `Дата проведения`, формат YYYY-MM-DD) |
| `regionCode` | string | Код підрозділу (той самий формат що в `login` — узгоджуйте з 1С) |
| `regionName` | string | Назва підрозділу (поле `Подразделение`) |
| `city` | string | Місто проведення (поле `Город`) |

**Логіка в 1С:**

1. Запит до справочника `Виды обучения` (на скриншоті — той що відкривається через "Направления")
2. Фільтр:
   - `Подразделение` відповідає `regionCode` (наприклад "ODS" → "Одеса")
   - `Дата проведения >= dateFrom`
   - **НЕ включати** навчання з ознакою `Отменен` (галочка на формі) або з заповненим `Перенос с` (якщо це означає скасування — уточнити)
3. Сортування: за `Дата проведения` ASC (ближчі за датою — спочатку)

**Важливо:**
- Метод викликається кожного разу коли менеджер відкриває форму планування (~раз на день). Швидкодія важлива, але не критична.
- Очікувана кількість записів — до 50 на регіон на місяць.
- Поле `Тип обучения` (з картки 1С) можна не повертати — у формі веб-додатку відображається `trainingName` + `date` + `regionName`.
- Якщо в регіоні немає навчань на майбутнє — повертати `trainings: []` (НЕ помилку). Веб-додаток обробить порожній список.
- Поле `Город` в 1С має формат "04.Одеса" (з префіксом коду) — повертати як є, веб-додаток сам обріже якщо треба.

### Чек-ліст тестів:

**login:**
- [ ] Менеджер: roleCode = "manager", managedUsers = []
- [ ] РМ: roleCode = "rm", managedUsers = [список логінів]
- [ ] Директор: roleCode = "director", managedUsers = [список РМ]
- [ ] Старі поля (login, role, auth) все ще присутні

**getClientsForPlanning:**
- [ ] Повертає всіх клієнтів менеджера
- [ ] У кожного є category
- [ ] purchases[] містить тільки бренди де були закупки
- [ ] lastPurchaseAmount — сума останнього документу, не загальна
- [ ] Клієнт без закупок — purchases: []

**getSalesFact:**
- [ ] totalFactUSD рахує ВСІ продажі менеджера (не тільки clientIds)
- [ ] 🆕 totalClientCount = COUNT(DISTINCT Контрагент) — кількість УНІКАЛЬНИХ покупців у місяці (не тільки з clientIds)
- [ ] 🆕 totalClientCount ≥ clients.length (бо клієнтів у факті може бути більше ніж переданих в clientIds)
- [ ] clients[] містить тільки контрагентів з переданого clientIds
- [ ] Кілька документів по одному клієнту/бренду — сумовані (один рядок у clients[], один в підрахунку totalClientCount)
- [ ] Пустий clientIds → totalFactUSD + totalClientCount є, clients порожній
- [ ] 🆕 `asOfDate` не передано → факт по останній день місяця `period` (не сьогодні!)

**getRegistryPlans:**
- [ ] Повертає плани по ВСІХ менеджерах/підрозділах
- [ ] Деталізація по сегментах

**getRegionData:**
- [ ] РМ бачить тільки свій підрозділ
- [ ] Директор бачить всі підрозділи
- [ ] totalPlan і totalFact вірно підсумовані
- [ ] 🆕 `asOfDate` обмежує факт по передану дату (порівняти: запит з `asOfDate` середини місяця має менший факт ніж без нього)

- [ ] 🆕 v2.3: `prevMonthAsOfDate` = останній день минулого місяця (наприклад "2026-03-31" для period "2026-04")
