## ⚠️ HIGH (тиждень 2-4)

### Логи витоку PII (4 знахідки)
- `src/app/api/onec/route.ts:220` — повний request payload з даними нових клієнтів
- `[ШАГ 2]` логує upstream 1С response (до 1KB) — leaks client data до Vercel logs
- Bitrix webhook логує base64 attachments
- Upstream 1С error message echo до клієнта → user enumeration

### Race & data-integrity
- `finalize` TOCTOU — SELECT-then-UPSERT не атомарні (`src/app/api/planning/finalize/route.ts:117`)
- `archive` DELETE лишає `planning_snapshots` orphaned (`src/app/api/archive/route.ts:67`)
- `user_id` без normalize (case/whitespace) → потенційні data silos (`src/app/api/planning/route.ts:173`)

### Безпека сесій
- Per-IP rate-limit для login — пароль можна підбирати через ротацію IP (`src/app/api/auth/login/route.ts:37`)
- JWT 24h без revocation — admin demotion діє лише після logout

### Frontend
- Open redirect через `notification.link` (XSS phishing) (`src/components/notifications/notifications-bell.tsx:112`)
- XSS у claim-detail (`dangerouslySetInnerHTML` + власний sanitizer що bypassable) (`src/components/claims/claim-detail-view.tsx:575`)
- Немає root Error Boundary → white screen на render error (`src/app/layout.tsx:42`)
- 2797 LOC у `clients-page.tsx`, 40+ inlined компонентів — складно змінювати

### Performance
- `ClientRow` component not memoized — 200+ rows re-render on every keystroke (`src/components/clients/clients-page.tsx:1494`)
- ClientsPage runs four parallel 1С Action chains (`src/lib/use-my-clients.ts:228`)

### A11y
- Form labels not programmatically associated з inputs (login + meeting form + claim form) (`src/components/login/login-form.tsx:72`)

## 🟡 MEDIUM (44 знахідки — групи)

1. **Sentry observability** — source maps не залиті, нема `beforeSend` PII scrubber, breadcrumbs з даними
2. **Webhook hardening** — timing-unsafe `!==`, опціональний dedup_key, unbounded JSON body, повторне використання старого секрету при ротації, manual sync скидає retry_count
3. **Performance** — `CategorySection` recomputes 4 filter passes без useMemo, 3 SWR-полери (30/60/15с) працюють навіть коли UI невидимий, відсутній індекс на `planning_snapshots(period_id, user_id, segment_code, block_type)`
4. **Director scope** — комент каже "read-only", фактично може писати фіналізації керованих менеджерів; не може редагувати їх зустрічі (404 при start/finish)
5. **Schema gaps** — нема FK `notifications.user_login → users`, нема CHECK на `users.role`, `dedup_key` UNIQUE надто широкий, мітки `archived_at=null` автоматично воскрешають soft-deleted рядки
6. **Hardcoded костилі** — `sdu@`, `itd@`, `rm.odessa@` без env-override (bus factor), дубльована date-parse у 11+ файлах, dead `FEATURES.PLANNING_DISABLED` (11 місць)
7. **API misc** — `x-api-key` через `!==` (timing oracle), `API_SECRET_KEY` fallback `dev-secret-key-change-me`, body size без ліміту, PostgREST column name не валідується
8. **A11y medium** — Dialog close button 28px, LoadingScreen без aria-live, FilterPill <44px, muted-foreground #7c879a fails WCAG AA contrast

## 🏗 Tech-stack — оцінка

**Що працює:** Next.js 16 App Router + Supabase REST через власний клієнт (через issue з Turbopack) — pragmatic вибір, документований у CLAUDE.md. Vercel auto-deploy швидкий і надійний. 1С через server-proxy — правильна архітектура.

**Що ризик:** **жодна core-таблиця не має RLS**, а service-role-key обходить будь-яку перевірку — фактично весь захист зведено до auth-guard у API-роутах. Якщо хтось пропустить `getSession()` у новому endpoint — це автоматично відкрита діра. RLS у `meetings`/`notifications` додано пізніше — треба ту саму модель розширити на planning-стек.

**Що еволюціонувати:** JWT-сесія 24h без revocation + per-instance rate-limit (працює тільки в межах одного warm Lambda) — на масштабі 21 менеджер працює, на 100+ почне сипатись. Альтернатива — Upstash Redis для rate-limit + session-blacklist.
