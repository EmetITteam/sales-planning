# ТЗ для 1С-розробника: Action `getActivityPlan` — план активностей менеджера

**Мета:** отримати з 1С документ **«План активностей»**, який ставиться **на менеджера × категорію клієнта** і задає, **скільки зустрічей (активностей) має бути виконано** по кожній категорії за місяць.

Це НЕ `getClientActivationPlan` (Action B). Різниця принципова:

| | Action B `getClientActivationPlan` | Цей Action `getActivityPlan` |
|---|---|---|
| Що планує | скільки **клієнтів** категорії **активувати** (довести до покупки) | скільки **зустрічей** треба **виконати** по категорії |
| Одиниця | клієнти (результат: купив/ні) | активності (процес: скільки зустрічей) |
| Документ 1С | «Планування активації бази клієнтів» | «План активностей» (уточнити точну назву) |

Формат виклику — стандартний (той самий web-сервіс, новий `action` у payload; Basic Auth + `X-Service-Token` як для решти дій):
```json
// Запит
{ "action": "getActivityPlan", "payload": { ... } }
// Відповідь успіх
{ "status": "success", "data": { ... } }
// Відповідь помилка
{ "status": "error", "message": "..." }
```

## Приклад документа у 1С (як ми його розуміємо)

```
Документ: «План активностей»
Номер: 000000XXX
Дата: 28.06.2026
Регіон: Київ
Менеджер: Мігашко Анна
Період: 01.07.2026 — 31.07.2026

Табична частина — план по категоріях:
| N | Категорія клієнта | Зустрічей (план) |
|---|---|---|
| 1 | Активный         | 20 |
| 2 | Спящий           | 10 |
| 3 | Потерянный       | 5  |
| 4 | Без закупок      | 8  |
| 5 | Новый            | 3  |
```

> ℹ️ **Обсяг на зараз:** документ планує **тільки зустрічі** (`planMeetings`). На майбутнє передбачаємо, що з'явиться і план по **дзвінках** — тому в схемі зарезервовано поле `planCalls`, але **зараз його НЕ заповнюємо** (можна опустити або `0`). Коли додасться план дзвінків — просто почнемо його наповнювати без зміни контракту.

## Запит

```json
{
  "action": "getActivityPlan",
  "payload": {
    "login": "headofsd@emet.in.ua",
    "period": "2026-07"
  }
}
```

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `login` | string | Так | Логін менеджера. Документ прив'язаний до менеджера — повертати лише його план. |
| `period` | string | Так | Місяць `YYYY-MM`. Шукаємо документ, чий період перетинається з цим місяцем. |

```json
{
  "status": "success",
  "data": {
    "period": "2026-07",
    "documentNumber": "000000841",
    "documentDate": "2026-06-28",
    "periodStart": "2026-07-01",
    "periodEnd": "2026-07-31",
    "region": "Київ",
    "managerLogin": "headofsd@emet.in.ua",
    "managerName": "Мігашко Анна",
    "plan": [
      { "category": "Активный",    "planMeetings": 20, "totalInCategory": 45  },
      { "category": "Спящий",      "planMeetings": 10, "totalInCategory": 130 },
      { "category": "Потерянный",  "planMeetings": 5,  "totalInCategory": 200 },
      { "category": "Без закупок", "planMeetings": 8,  "totalInCategory": 150 },
      { "category": "Новый",       "planMeetings": 3,  "totalInCategory": 12  }
    ]
  }

```

### Поля шапки

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `period` | string | Так | Як у запиті — `YYYY-MM`. |
| `documentNumber` | string \| null | Так | Номер документа. `null` якщо документа нема. |
| `documentDate` | string \| null | Так | Дата документа `YYYY-MM-DD`. `null` якщо нема. |
| `periodStart` | string \| null | Так | Початок періоду плану `YYYY-MM-DD`. |
| `periodEnd` | string \| null | Так | Кінець періоду плану `YYYY-MM-DD`. |
| `region` | string \| null | Так | Регіон менеджера. |
| `managerLogin` | string | Так | Логін як у запиті (для валідування). |
| `managerName` | string \| null | Ні | ПІБ менеджера (зручно для звіту). |
| `plan` | ActivityPlanRow[] | Так | Рядки табличної частини. Порожній `[]` якщо документа нема. |

### Поля `ActivityPlanRow`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `category` | string | Так | Категорія клієнта в **оригінальному RU-формулюванні 1С**: `Активный` / `Спящий` / `Потерянный` / `Без закупок` / `Новый`. (Sales Planning має мапінг RU→UA.) |
| `planMeetings` | number | Так | **Скільки зустрічей** має бути виконано по цій категорії за період. Ціле число. Головне (і поки що єдине) поле плану. |
| `planCalls` | number | Ні | **Зарезервовано на майбутнє** — план дзвінків. **Зараз не заповнювати** (опустити або `0`). Контракт готовий, наповнимо коли з'явиться план дзвінків. |
| `totalInCategory` | number | Ні | Скільки всього клієнтів цієї категорії у менеджера зараз (справково, для «база vs план»). Ціле. |

## Логіка в 1С

1. Знайти **актуальний** документ «План активностей»:
   - `Менеджер.Логін == login`
   - період документа перетинається з місяцем: `початок ≤ останній_день_місяця AND кінець ≥ перший_день_місяця`
   - якщо документів кілька — взяти останній за `Дата`.
2. Якщо знайдено — повернути шапку + розгорнути табличну частину у `plan[]`.
3. Якщо документа нема — повернути `documentNumber=null, …=null, plan=[]` (**НЕ помилка**).

## Edge cases

- Документ є, табличка пуста → `plan: []`, шапка заповнена. Не помилка.
- `period` у майбутньому → шукати все одно (менеджер міг заповнити наперед).
- `period` минулий → підтримувати без обмежень (потрібна історія план vs факт).
- Менеджера нема у 1С → `{ "status": "error", "message": "manager not found" }`.

## Performance

- Ціль: < 200мс. Частота: 1–2 рази на сесію. Кешуємо на 5хв.

## ⭐ Action 2 (обов'язковий): `getActivityPlansBulk` — план по ВСІХ менеджерах

Реалізувати **разом з першим** (не пізніше). Головний сценарій — **звід «план активностей vs факт» по всіх менеджерах** (директор/РОП бачить хто виконує норму зустрічей). Викликати `getActivityPlan` по кожному логіну окремо — повільно, тому потрібен bulk за зразком `getRegistryPlans` (Action 4):

```json
// Запит
{ "action": "getActivityPlansBulk", "payload": { "dateFrom": "2026-07-01", "dateTo": "2026-07-31" } }
```
```json
// Відповідь: плоский масив, по одному рядку на (менеджер × категорія)
{
  "status": "success",
  "data": {
    "plans": [
      { "managerLogin": "headofsd@emet.in.ua", "managerName": "Мігашко Анна", "region": "Київ",
        "category": "Активный", "planMeetings": 20 },
      { "managerLogin": "sm.kiev3@emet.in.ua", "managerName": "Селиванова Вікторія", "region": "Київ",
        "category": "Спящий", "planMeetings": 12 }
      // … `planCalls` — так само зарезервовано, зараз не заповнюємо
    ]
  }
}
```
Логіка та сама, лише без фільтра по `login` — усі актуальні документи, чий період перетинається з `[dateFrom; dateTo]`. Поля рядка — ті самі, що в `ActivityPlanRow`, плюс `managerLogin` / `managerName` / `region`.

## Як Sales Planning це використає (для довідки)

**Факт ми рахуємо самі, 1С дає лише ПЛАН:**
- Факт зустрічей — з нашої таблиці `meetings` (`status='done'`, `date` у періоді, `client_category`), групуємо по менеджеру × категорії.
- Факт дзвінків (якщо плануються) — поки лише boolean через `checkActivities` (Action 7); повноцінних подій дзвінків у нас нема (окреме питання до 1С).

**UI — таблиця «Активності: план vs факт» по категоріях:**

| Категорія | План зустрічей | Виконано | % | 
|---|---|---|---|
| Активні | 20 | 14 | 70% |
| Сплячі | 10 | 6 | 60% |
| Втрачені | 5 | 2 | 40% |

З bulk-варіантом — той самий звід агрегуємо по менеджерах → регіонах → компанії (як зараз працює план продажів).

```ts
// орієнтовна інтеграція
const { data } = useOneCData('getActivityPlan', login && month ? { login, period: month } : null);
// data.plan[] → merge з фактом зустрічей з Supabase по category
```

## ✅ Обсяг узгоджено (ITD, 2026-07-10)

- **Тільки зустрічі** зараз (`planMeetings`). `planCalls` — зарезервовано в контракті, поки НЕ заповнюємо.
- **Обидва action** реалізувати разом: `getActivityPlan` (per-manager) + `getActivityPlansBulk` (усі менеджери).
- Одиниця — **кількість зустрічей** (активностей), не кількість клієнтів.
