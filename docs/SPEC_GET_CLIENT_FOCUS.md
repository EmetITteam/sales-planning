**Куди використовується:** Картка клієнта на сторінці `/clients` — CRM-режим менеджера для підготовки до дзвінка.

## Загальне

Дві нові HTTP-дії додаються до існуючого 1С web-сервісу. Виклик через ту саму endpoint-у що `getClientReport`/`getRegionData` тощо — просто новий `action` у payload. Авторизація — Basic Auth + `X-Service-Token` (як для решти дій).

Формат запиту/відповіді — стандартний:
```json
// Запит
{ "action": "<actionName>", "payload": { ... } }
// Відповідь успіх
{ "status": "success", "data": { ... } }
// Відповідь помилка
{ "status": "error", "message": "..." }
```

# Action A: `getClientFocus`

**Призначення:** Повернути **список активних фокусів** для масиву клієнтів. Фокус — запис з регістра відомостей у 1С (назву регістра 1С-розробник знає). На одного клієнта може бути **кілька активних фокусів одночасно**. Фокус має дату початку і опційно дату закінчення.

## A.1 — Запит

```json
{
  "action": "getClientFocus",
  "payload": {
    "login": "manager.dnepr@emet.in.ua",
    "clientIds": ["C001", "C002", "C003"]
  }
}
```

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `login` | string | ✅ | Логін менеджера. Фокус прив'язаний до менеджера — повертати лише ті, що виставлені для цього менеджера. |
| `clientIds` | string[] | ✅ | Масив `Код` контрагентів. Від 1 до ~500 елементів. |

## A.2 — Відповідь

```json
{
  "status": "success",
  "data": {
    "focuses": [
      {
        "clientId": "C001",
        "items": [
          { "focusName": "У фокусі: Neuronox",   "since": "2026-04-15", "validUntil": "2026-07-15" },
          { "focusName": "Реактивація",          "since": "2026-05-01", "validUntil": null }
        ]
      },
      {
        "clientId": "C002",
        "items": [
          { "focusName": "Новий — онбординг", "since": "2026-05-20", "validUntil": "2026-08-20" }
        ]
      },
      {
        "clientId": "C003",
        "items": []
      }
    ]
  }
}
```

### Поля `focuses[]`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `clientId` | string | ✅ | Код контрагента (як у запиті). |
| `items` | Focus[] | ✅ | Масив активних фокусів. Порожній `[]` якщо клієнт ні у одному фокусі. **Не пропускати** клієнта зі списку — повертаємо рядок з пустим `items` для consistency `focuses.length === clientIds.length`. |

### Поля `Focus`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `focusName` | string | ✅ | Текст з регістра 1С як є. Це українські назви які менеджери вже використовують — переносимо as-is. |
| `since` | string | ✅ | Дата встановлення фокусу, `YYYY-MM-DD`. |
| `validUntil` | string \| null | ✅ | Дата закінчення фокусу `YYYY-MM-DD`. `null` — безстроковий. |

## A.3 — Логіка в 1С

1. Звернутись до регістра відомостей (назва відома 1С-розробнику).
2. Фільтри:
   - `Контрагент IN clientIds`
   - `Менеджер == login`
   - Тільки **активні на сьогодні**: `Дата ≤ TODAY AND (ДатаЗакінчення IS NULL OR ДатаЗакінчення ≥ TODAY)`
3. Сгрупувати по `Контрагент` → отримати масив `items[]` (може бути 0, 1 або більше записів).
4. Для **кожного** `clientId` зі вхідного списку — повертати рядок (з пустим `items` якщо фокусів нема).

## A.4 — Edge cases

- `clientIds` порожній → `focuses: []` (НЕ помилка).
- `clientIds` довший ~500 — або 1С обробляє за один запит, або скажіть і ми чанкуємо на нашій стороні (як для `getSalesFact` 400-ліміту).
- Клієнт існує, але прив'язаний до іншого менеджера → повертати з пустим `items` (privacy: не показуємо чужі фокуси).
- Якщо фокус у регістрі є але вже скінчився (`validUntil < TODAY`) — НЕ повертати.

## A.5 — Performance

- Цільова швидкодія: **< 300мс на запит** для ~500 clientIds.
- Очікувана частота: **1 раз на сесію** (відкриття `/clients`) + кеш у SWR на 60-300с.

# Action B: `getClientActivationPlan`

**Призначення:** Повернути **план активації клієнтів по категоріях** для менеджера за заданий період (місяць). Джерело — документ/регістр **«Планування активації бази клієнтів»** у 1С. Менеджер у документі вказує, скільки клієнтів з кожної категорії (Спящий / Потерянный / Без закупок / тощо) він планує активувати протягом місяця. У документі також зберігається довідкове «Кількість клієнтів у категорії» — щоб видно було база vs план.

**Приклад документа у 1С:**
```
Номер: 000000527
Дата: 29.04.2026
Регіон: Харків
Менеджер: Тесленко Марія
Період реєстрації: 01.05.2026 — 31.05.2026

Активності по категоріях:
| N | Категорія клієнта | План | Клієнтів у категорії |
|---|---|---|---|
| 1 | Спящий           | 3    | 36   |
| 2 | Потерянный       | 4    | 208  |
| 3 | Без закупок      | 3    | 148  |
```

## B.1 — Запит

```json
{
  "action": "getClientActivationPlan",
  "payload": {
    "login": "manager.kharkiv@emet.in.ua",
    "period": "2026-05"
  }
}
```

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `login` | string | ✅ | Логін менеджера. Документ прив'язаний до менеджера — повертати лише його план. |
| `period` | string | ✅ | Місяць у форматі `YYYY-MM`. Шукаємо документ де `Період реєстрації` перетинається з цим місяцем (або точно дорівнює). |

## B.2 — Відповідь

```json
{
  "status": "success",
  "data": {
    "period": "2026-05",
    "documentNumber": "000000527",
    "documentDate": "2026-04-29",
    "periodStart": "2026-05-01",
    "periodEnd": "2026-05-31",
    "region": "Харків",
    "activations": [
      { "category": "Спящий",      "planCount": 3, "totalInCategory": 36  },
      { "category": "Потерянный",  "planCount": 4, "totalInCategory": 208 },
      { "category": "Без закупок", "planCount": 3, "totalInCategory": 148 }
    ]
  }
}
```

### Поля відповіді

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `period` | string | ✅ | Як у запиті — `YYYY-MM`. |
| `documentNumber` | string \| null | ✅ | Номер документа у 1С (для довідки). `null` якщо документ не заведено. |
| `documentDate` | string \| null | ✅ | Дата документа `YYYY-MM-DD`. `null` якщо нема. |
| `periodStart` | string \| null | ✅ | Початок періоду реєстрації `YYYY-MM-DD`. `null` якщо нема. |
| `periodEnd` | string \| null | ✅ | Кінець періоду реєстрації. |
| `region` | string \| null | ✅ | Регіон менеджера як у документі. |
| `activations` | ActivationRow[] | ✅ | Масив рядків з табличної частини документа. Може бути порожній якщо документу нема. |

### Поля `ActivationRow`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `category` | string | ✅ | Категорія клієнта в оригінальному формулюванні з 1С (`Спящий` / `Потерянный` / `Без закупок` / `Активный` / `Новый`). Sales Planning має мапінг RU→UA (вже працює для Action 5). |
| `planCount` | number | ✅ | Скільки клієнтів цієї категорії менеджер планує активувати у періоді. Цілочислове. |
| `totalInCategory` | number | ✅ | Скільки всього клієнтів цієї категорії у менеджера зараз (довідкове поле з документа). Цілочислове. |

## B.3 — Логіка в 1С

1. Знайти **актуальний** документ «Планування активації бази клієнтів»:
   - `Менеджер.Логін == login`
   - `Період реєстрації початок ≤ period_last_day AND Період реєстрації кінець ≥ period_first_day` (перетин з місяцем)
   - Якщо документів кілька (рідкісний випадок) — взяти останній за `Дата`.
2. Якщо документ знайдено:
   - Повернути шапку (`documentNumber`, `documentDate`, `periodStart`, `periodEnd`, `region`)
   - Розгорнути табличну частину «Активності по категоріях» → `activations[]`
3. Якщо документу нема:
   - Повернути `documentNumber=null, …=null, activations=[]` (НЕ помилка).

## B.4 — Edge cases

- Документ є але табличка пуста → `activations: []`, шапка заповнена. Не помилка.
- `period` у майбутньому → шукати все одно (менеджер міг заздалегідь заповнити).
- `period` старший — підтримуємо без обмежень (історія планів цінна).
- Якщо менеджера нема у 1С → `status: error, message: "manager not found"`.

## B.5 — Performance

- Цільова швидкодія: **< 200мс**.
- Очікувана частота: **1-2 рази на сесію** менеджера (відкриття дашборду + перерахунок). Кешуємо на 5хв.

# Action C: Розширення `getManagerClients` — додати `isReserved`

**Призначення:** Поточна дія `getManagerClients` повертає базову інформацію по клієнтах менеджера. Потрібно додати **`isReserved: boolean`** — позначка чи клієнт у «Резерві» (на нього менеджер не звертає уваги).

**Чому НЕ через properties:** `properties` зараз доступне тільки через `getClientReport` (per-client lazy). Sales Planning потребує знати «резервність» клієнта **upfront для всього списку 481 клієнтів** — для фільтра-pills і окремого розділу у списку. Per-client запити не масштабуються.

## C.1 — Зміна формату відповіді `getManagerClients`

**Було** (поточний формат `getManagerClients`):
```json
{
  "clients": [{
    "ClientID": "...",
    "ClientName": "...",
    "ClientCategory": "...",
    "ClientAddress": "...",
    "Phone": "..."
  }]
}
```

**Стало** (додати одне нове поле):
```json
{
  "clients": [{
    "ClientID": "...",
    "ClientName": "...",
    "ClientCategory": "...",
    "ClientAddress": "...",
    "Phone": "...",
    "isReserved": false
  }]
}
```

> **Примітка:** `managerName` / `isMine` у `getManagerClients` нема (вони лише у `findClient` бо там результат включає чужих менеджерів). У `getManagerClients` усі клієнти — свої, тому ці поля не потрібні.

## C.2 — Логіка `isReserved`

- `true` — клієнт має у `properties` (або там де воно зберігається у 1С) категорію/прапор «Резерв»
- `false` — у решті випадків

## C.3 — Що використовується у Sales Planning

- Окрема hero-картка з лічильником резерв-клієнтів
- Окрема секція у списку (за замовч згорнута)
- Tag `[Резерв]` поряд з category-chip у рядку клієнта
- Включаються у загальний counter «База клієнтів» (це теж база — але виключені з планування і активації)

## C.4 — Інші точки інтеграції

- `findClient` — додати те саме поле у його response (для глобального пошуку)
- `getClientReport` — `properties[]` лишається як є (для повного детального view)

## Sales Planning — як використовуватиме (для довідки)

```ts
// src/lib/use-my-clients.ts (extension)
const focusPayload = login && clientIds.length > 0
  ? { login, clientIds }
  : null;
const { data: focusRes } = useOneCData('getClientFocus', focusPayload);

const focusByClient = useMemo(() => {
  const out: Record<string, Focus[]> = {};
  for (const f of focusRes?.focuses ?? []) {
    out[f.clientId] = f.items;
  }
  return out;
}, [focusRes]);
```

UI: у рядку клієнта поряд з category-chip — невеликі chip-и з кожним активним фокусом. Колір — нейтральний (бежевий/teal). В expanded view — окремий блок «Фокус(и) · N» з більшою деталізацією (включно з `since`/`validUntil`).
