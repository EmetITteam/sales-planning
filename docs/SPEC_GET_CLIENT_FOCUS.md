---

## 1. Мета

Sales Planning має у себе сторінку «Мої клієнти» (CRM-режим менеджера). Картка кожного клієнта вже показує:
- Базову інфу (телефон, освіта, документи, властивості)
- План × Факт цього місяця по брендах
- 3-міс історію купівель
- Події (зустрічі, дзвінки, семінари)

Треба додати **«У якому фокусі зараз перебуває клієнт»** — інформація з регістра відомостей 1С (TBD: уточнити назву регістра).

**Чому окремою дією, а не у `getClientReport`:**
- `getClientReport` — важка дія (3 міс історія + всі події). Кешуємо її lazy при кліку.
- Фокуси потрібні для **усіх клієнтів менеджера одразу** (підказки на списку, не лише у відкритій картці).
- Окрема bulk-дія — швидко, легко, кешуємо незалежно.

## 3. Запит (формат як інші дії)

```json
{
  "action": "getClientFocus",
  "payload": {
    "login": "manager.dnepr@emet.in.ua",
    "clientIds": ["C001", "C002", "C003"]
  }
}
```

### Параметри

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `login` | string | ✅ | Логін менеджера (як у Action 1-7). 🚩 **TBD:** чи прив'язаний фокус до менеджера (різні менеджери — різний фокус того ж клієнта), чи це shared атрибут клієнта? Якщо shared — `login` тільки для security/audit, не для фільтра. |
| `clientIds` | string[] | ✅ | Масив `Код` клієнтів. Може бути від 1 до ~500 елементів (у менеджера буває до 481 клієнта). Якщо 500+ — додатково обговоримо. |

## 5. Логіка в 1С

1. Брати **актуальний** запис з регістра відомостей (TBD: уточнити назву регістру — найімовірніше «Фокус клієнтів» / «ФокусыКлиентов» / «УстановкаФокусаКонтрагентов»).
2. Фільтри:
   - `Контрагент IN clientIds`
   - Якщо фокус прив'язаний до менеджера: `Менеджер == login` (TBD підтвердити)
   - Тільки **поточний** активний запис (на сьогоднішню дату): `Період ≤ TODAY AND (ДатаЗакінчення IS NULL OR ДатаЗакінчення ≥ TODAY)`
3. Для кожного `clientId` з `clientIds` — повертати ОДИН рядок:
   - Якщо знайдено активний фокус → заповнити поля
   - Якщо клієнт не у фокусі → повертати рядок з `focusCode=null`, `focusName=null`, інші null (НЕ пропускати клієнта — UI потребує `focuses.length === clientIds.length` для консистентності).

## 7. Edge cases

- `clientIds` порожній → `focuses: []` (НЕ помилка).
- `clientIds` довший 500 → 1С відповідає 200 OK з обмеженням, або ми чанкуємо на стороні Sales Planning (як зробили для `getSalesFact` 400-ліміту). Скажіть який підхід зручніший.
- Клієнт існує у 1С, але фокусу не виставлено → рядок з `focusCode: null`, не пропускати.
- Клієнт існує у 1С, але прив'язаний до іншого менеджера (`isMine: false`) → 🚩 **TBD:** повертати фокус, чи `null`? (privacy чи інформативність)

## 8. Важливо

- Виклик через існуючий HTTP-сервіс 1С — те ж API endpoint, що `getClientReport` тощо. Просто новий `action` у payload.
- Auth — Basic Auth + `X-Service-Token` (як зараз для решти дій).
- Швидкодія: цільова — **< 300мс на запит** для ~500 clientIds. Якщо запит важкий — потрібен індекс по (Контрагент, Період).
- Очікувана частота: **раз при відкритті сторінки `/clients`** + у кешу SWR на 60-300с.

## 9. Sales Planning використовуватиме

```ts
// hook src/lib/use-my-clients.ts (extension)
const focusPayload = login && clientIds.length > 0
  ? { login, clientIds }
  : null;
const { data: focusRes } = useOneCData('getClientFocus', focusPayload);
const focusByClient = useMemo(() => {
  const out: Record<string, FocusInfo> = {};
  for (const f of focusRes?.focuses ?? []) {
    if (f.focusCode) out[f.clientId] = f;
  }
  return out;
}, [focusRes]);
```

UI: невеликий chip у рядку клієнта поряд з category-chip типу `[У фокусі: Neuronox]`. Або в expanded view — у блоку «Інформація по клієнту».
