## Загальне

Дві нові HTTP-дії додаються до існуючого 1С web-сервісу. Виклик через ту саму endpoint-у що `getClientReport`/`getRegionData` тощо — просто новий `action` у payload. Авторизація — Basic Auth + `X-Service-Token` (як для решти дій).

Формат запиту/відповіді — стандартний:
```json
// Запит
{ "action": "<actionName>", "payload": { ... } }
// Відповідь успіх
{ "status": "success", "data": { ... } }
// Відповідь помилка
{ "status": "error", "message": "..." }
```

# Action A: `getClientFocus`

**Призначення:** Повернути **список активних фокусів** для масиву клієнтів. Фокус — запис з регістра відомостей у 1С (назву регістра 1С-розробник знає). На одного клієнта може бути **кілька активних фокусів одночасно**. Фокус має дату початку і опційно дату закінчення.

## A.1 — Запит

```json
{
  "action": "getClientFocus",
  "payload": {
    "login": "manager.dnepr@emet.in.ua",
    "clientIds": ["C001", "C002", "C003"]
  }
}
```

| Параметр | Тип | Обов'язковий | Опис |
|---|---|:---:|---|
| `login` | string | ✅ | Логін менеджера. Фокус прив'язаний до менеджера — повертати лише ті, що виставлені для цього менеджера. |
| `clientIds` | string[] | ✅ | Масив `Код` контрагентів. Від 1 до ~500 елементів. |

## A.2 — Відповідь

```json
{
  "status": "success",
  "data": {
    "focuses": [
      {
        "clientId": "C001",
        "items": [
          { "focusName": "У фокусі: Neuronox",   "since": "2026-04-15", "validUntil": "2026-07-15" },
          { "focusName": "Реактивація",          "since": "2026-05-01", "validUntil": null }
        ]
      },
      {
        "clientId": "C002",
        "items": [
          { "focusName": "Новий — онбординг", "since": "2026-05-20", "validUntil": "2026-08-20" }
        ]
      },
      {
        "clientId": "C003",
        "items": []
      }
    ]
  }
}
```

### Поля `focuses[]`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `clientId` | string | ✅ | Код контрагента (як у запиті). |
| `items` | Focus[] | ✅ | Масив активних фокусів. Порожній `[]` якщо клієнт ні у одному фокусі. **Не пропускати** клієнта зі списку — повертаємо рядок з пустим `items` для consistency `focuses.length === clientIds.length`. |

### Поля `Focus`

| Поле | Тип | Обов'язкове | Опис |
|---|---|:---:|---|
| `focusName` | string | ✅ | Текст з регістра 1С як є. Це українські назви які менеджери вже використовують — переносимо as-is. |
| `since` | string | ✅ | Дата встановлення фокусу, `YYYY-MM-DD`. |
| `validUntil` | string \| null | ✅ | Дата закінчення фокусу `YYYY-MM-DD`. `null` — безстроковий. |

## A.3 — Логіка в 1С

1. Звернутись до регістра відомостей (назва відома 1С-розробнику).
2. Фільтри:
   - `Контрагент IN clientIds`
   - `Менеджер == login`
   - Тільки **активні на сьогодні**: `Дата ≤ TODAY AND (ДатаЗакінчення IS NULL OR ДатаЗакінчення ≥ TODAY)`
3. Сгрупувати по `Контрагент` → отримати масив `items[]` (може бути 0, 1 або більше записів).
4. Для **кожного** `clientId` зі вхідного списку — повертати рядок (з пустим `items` якщо фокусів нема).

## A.4 — Edge cases

- `clientIds` порожній → `focuses: []` (НЕ помилка).
- `clientIds` довший ~500 — або 1С обробляє за один запит, або скажіть і ми чанкуємо на нашій стороні (як для `getSalesFact` 400-ліміту).
- Клієнт існує, але прив'язаний до іншого менеджера → повертати з пустим `items` (privacy: не показуємо чужі фокуси).
- Якщо фокус у регістрі є але вже скінчився (`validUntil < TODAY`) — НЕ повертати.

## A.5 — Performance

- Цільова швидкодія: **< 300мс на запит** для ~500 clientIds.
- Очікувана частота: **1 раз на сесію** (відкриття `/clients`) + кеш у SWR на 60-300с.

## Sales Planning — як використовуватиме (для довідки)

```ts
// src/lib/use-my-clients.ts (extension)
const focusPayload = login && clientIds.length > 0
  ? { login, clientIds }
  : null;
const { data: focusRes } = useOneCData('getClientFocus', focusPayload);

const focusByClient = useMemo(() => {
  const out: Record<string, Focus[]> = {};
  for (const f of focusRes?.focuses ?? []) {
    out[f.clientId] = f.items;
  }
  return out;
}, [focusRes]);
```

UI: у рядку клієнта поряд з category-chip — невеликі chip-и з кожним активним фокусом. Колір — нейтральний (бежевий/teal). В expanded view — окремий блок «Фокус(и) · N» з більшою деталізацією (включно з `since`/`validUntil`).
