# Вбудовування Sales Planning у 1С — спека для розробника 1С

**Мета:** Менеджер відкриває «Поле HTML-документа» (або Web Browser Control) у 1С — бачить дашборд Sales Planning без окремого логіну. SSO через 1С.

**Альтернатива:** замість вбудовування — кнопка у меню 1С яка відкриває `https://sales-planning-lyart.vercel.app` у системному браузері. Простіше, але втрачаємо «вбудованість». Розглянути обидва варіанти.

---

## Варіант A: повноцінне вбудовування з SSO

### Що робить 1С

При відкритті форми з вбудованим Sales Planning:
1. Згенерувати **token** (один раз на сесію або на кожне відкриття).
2. Сформувати URL з token:
   ```
   https://sales-planning-lyart.vercel.app/embed?login={user_login}&t={token}&exp={unix_ts_expires}
   ```
3. Завантажити цей URL у HTML-поле / Web Browser Control.

### Token формат

**HMAC-SHA256** підпис над `login + exp` зі shared secret (env var на нашому Vercel `EMBED_SHARED_SECRET`):

```
message = `${login}|${exp}`           // exp = Unix timestamp коли token згоріє
token = base64url( HMAC_SHA256(EMBED_SHARED_SECRET, message) )
```

**Приклад на 1С (псевдокод):**
```bsl
&НаСервере
Функция ЗгенеруватиТокен(Логін)
    КінецьДії = Формат(ПоточнаДата() + 3600, "ЧДЦ=0; ЧГ=0"); // +1 година
    Повідомлення = Логін + "|" + КінецьДії;
    Секрет = Константи.ВбудованийСекретSP.Получить();          // EMBED_SHARED_SECRET
    Токен = ОбчислитиHMACSHA256(Секрет, Повідомлення);
    Повернення Base64URL(Токен);
КінецьФункції
```

(Точний синтаксис під 1С УТП підбере програміст. Стандартні методи криптографії в 1С: `КриптоMenedger`, `XDTOСеріалізатор` тощо.)

### Що робить наш веб-додаток (наша робота, не твоя)

1. Реєструємо новий route `/embed`:
   - Читає `?login=...&t=...&exp=...` з URL
   - Перевіряє: `exp` не в минулому
   - Перевіряє HMAC: `HMAC_SHA256(EMBED_SHARED_SECRET, login + "|" + exp) === token`
   - Якщо OK — викликає 1С Action 1 (`login`) для отримання повного UserSession (роль, регіон, fullName) **АБО** додаємо новий 1С action `getUserByLogin` що повертає UserSession без пароля
   - Ставить HttpOnly cookie `sp_session` (як після звичайного логіну)
   - Редірект на `/`
2. Додаємо env `EMBED_SHARED_SECRET` на Vercel (`openssl rand -hex 32`).
3. Передаємо secret 1С розробнику для збереження у 1С (як константа або зашифровано).

### Повний flow

```
Менеджер відкриває форму у 1С
        ↓
1С генерує token (HMAC секретом, 1 година TTL)
        ↓
1С будує URL: /embed?login=ivanov@emet.com&t=xxx&exp=1714576800
        ↓
Завантажує URL у HTML-поле
        ↓
Наш /embed:
  • verify HMAC + exp
  • fetch UserSession з 1С (Action 1 без password АБО новий action)
  • set sp_session cookie
  • redirect /
        ↓
Дашборд відкривається — менеджер не вводив пароль
```

### Плюси/мінуси Варіант A

✅ **Плюси:**
- Менеджер не вводить пароль ще раз
- 1С є основним інструментом — все «в одному вікні»
- Token-based SSO працює і для preview deploys

❌ **Мінуси:**
- Потрібен новий 1С action `getUserByLogin` (без пароля) АБО реюз Action 1 з прапорцем «системний виклик»
- Shared secret — треба зберігати у 1С безпечно
- HTML-поле в 1С УТП на десктопі базується на старому WebView (Internet Explorer 11 на старих версіях, Edge на новіших) — деякі CSS-фічі можуть виглядати інакше

## Варіант B: просто кнопка з зовнішнім посиланням (швидкий)

**1С робота:** додати в меню/тулбар кнопку «Sales Planning», клік відкриває `https://sales-planning-lyart.vercel.app` у системному браузері.

**Плюси:**
- 30 хвилин роботи 1С-розробника
- Жодного нового API не треба
- Браузер коректно обробить наш PWA + cookie

**Мінуси:**
- Окреме вікно браузера (не в 1С)
- Менеджер вводить пароль один раз на 24 год (cookie TTL)

Для пілоту цього достатньо. Варіант A — пізніше якщо буде попит.

---

## Технічні питання для уточнення з 1С-розробником

1. **Яка версія 1С УТП?** (базується на цьому: HTML-поле це WebKit чи IE-engine)
2. **Чи легко додати глобальну константу/секрет?** (для EMBED_SHARED_SECRET)
3. **Чи має 1С функцію HMAC-SHA256 з коробки?** (більшість 1С 8.3+ — так, через `КриптоMenedger`)
4. **Чи є можливість логувати token у 1С** (для debug якщо щось не сходиться)?

---

## Коли робити Варіант A

Не до Action 5 wiring + рішення про `users.login` PK migration. Інакше робимо двічі. Логіка:

1. **Зараз → пілот:** Варіант B (просто кнопка) + PWA встановлення на робочому столі
2. **Після Action 5 + RLS:** Варіант A (SSO embed)
