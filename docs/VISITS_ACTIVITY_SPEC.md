## 1. Мета

Окремий борд, де директор / РОП бачить **план vs факт візитів (зустрічей)** по менеджерах: скільки візитів заплановано по кожній категорії клієнта і скільки фактично виконано. Розбивка регіон → менеджер, drill-down до контрагентів/дат, коментар менеджера по кожному візиту.

**1С дає лише ПЛАН. Факт рахуємо самі** (візити ведуться в нашому застосунку, лежать у `meetings`). 1С-метод візитів потрібен для **історії + звірки**.

## 3. Дані

### Факт візитів — вже в БД ✅
Таблиця `meetings` (нове сховище не потрібне). Візит = рядок зі `status='done'`.
Поля для борду: `manager_login`, `client_id_1c`, `client_name`, `client_category`, `date`, `comment`, `purpose`.

### План — з 1С ⏳
`getActivityPlan` / `getActivityPlansBulk` (контракти §7). План на менеджера × категорію: скільки візитів має бути.

---

## 4. Бізнес-правила підрахунку (КЛЮЧОВЕ)

Для кожного менеджера × категорія рахуємо **три числа**:

1. **Виконано плану (по УНІКАЛЬНИХ клієнтах)** — скільки **різних** контрагентів категорії отримали ≥1 виконаний візит у періоді.
   - 10 візитів до одного клієнта = **1** закритий план-клієнт.
   - `COUNT(DISTINCT client_id_1c) WHERE category=X AND status=done AND date in period`.
   - Саме це порівнюємо з планом → **% виконання**.
2. **Всього візитів** — сирий `COUNT(*)` (10 візитів = 10). Довідково.
3. **План** — `planMeetings` з 1С для категорії.

**% виконання = унікальні_клієнти / план × 100** (НЕ всього_візитів / план).
Агрегація: менеджер → регіон → компанія (сума планів, сума унікальних, сума візитів; % з сум).

## 5. Доступи (ОНОВЛЕНО — переиспользуємо існуюче)

Окрема сторінка `/admin/visits`. Гейтинг — **той самий патерн, що у Тижневому звіті** (не нова таблиця):

- **РМ (роль `rm`)** — авто по **своєму регіону** (+ мульти-регіон/гранти через `resolveRegionOverrides`, як у weekly-report). Дані інших регіонів приховані.
- **Повний борд (компанія)** — `admin` / `director` / `canViewCompanyOverview===true` / страт-логіни. Той самий гейт, що вже керує «Оглядом компанії» — **окрему `visits_permissions` НЕ робимо** (керується існуючим `/admin/company-overview-permissions`).
- Звичайний менеджер без гранту — нема доступу.

Пункт меню — в account-dropdown (як «Тижневий звіт»), гейт `canViewWeeklyReport`-стилю.

## 6. Пайплайн даних (ОНОВЛЕНО — образець sales-sync)

**Архітектура ідентична вже працюючому `sync-sales`** (`getSalesLineItems`→`sales`→крон). Копіюємо:

| Період | Джерело | Механізм |
|---|---|---|
| **Історія (минулі місяці цього року)** | `getVisitsBulk` (1С) | одноразовий backfill → `meetings` (upsert по `docId`=`legacy_1c_id`) |
| **Закриті місяці далі** | `getVisitsBulk` | крон помісячно доливає (як sync-sales тягне попередній місяць у перші 10 днів) |
| **Поточний місяць** | наша `meetings` (live) + звірка `getVisitsBulk` | факт уже у нас; крон періодично звіряє, чи не змінилось у 1С |

Реалізація: `sales-sync.ts` → аналог `visits-sync.ts`; крон `/api/cron/sync-visits` (як `sync-sales`); idempotent upsert по `docId`; мапінг категорій RU→UA (є `mapClientCategory`). Guard від wipe при порожньому-успішному респонсі — як у sales-sync.

## 7. 1С Actions (повні контракти)

Формат — стандартний web-сервіс (Basic Auth + `X-Service-Token`; новий `action` у payload):
```json
{ "action": "...", "payload": { ... } }   →   { "status": "success", "data": { ... } } | { "status": "error", "message": "..." }
```

### 7.1 `getActivityPlan` — план активностей менеджера

Документ 1С «План активностей» ставиться **на менеджера × категорію** і задає **скільки зустрічей** треба виконати за місяць. Це НЕ `getClientActivationPlan` (той планує активацію КЛІЄНТІВ, а цей — кількість ЗУСТРІЧЕЙ).

**Запит:**
```json
{ "action": "getActivityPlan", "payload": { "login": "headofsd@emet.in.ua", "period": "2026-07" } }
```
| Параметр | Тип | Обов. | Опис |
|---|---|:---:|---|
| `login` | string | Так | Логін менеджера. |
| `period` | string | Так | Місяць `YYYY-MM`. Документ, чий період перетинається з місяцем. |

**Відповідь:**
```json
{ "status": "success", "data": {
  "period": "2026-07", "documentNumber": "000000841", "documentDate": "2026-06-28",
  "periodStart": "2026-07-01", "periodEnd": "2026-07-31",
  "region": "Київ", "managerLogin": "headofsd@emet.in.ua", "managerName": "Мігашко Анна",
  "plan": [
    { "category": "Активный",    "planMeetings": 20, "totalInCategory": 45  },
    { "category": "Спящий",      "planMeetings": 10, "totalInCategory": 130 },
    { "category": "Потерянный",  "planMeetings": 5,  "totalInCategory": 200 },
    { "category": "Без закупок", "planMeetings": 8,  "totalInCategory": 150 },
    { "category": "Новый",       "planMeetings": 3,  "totalInCategory": 12  }
  ]
} }
```
**Поля шапки:** `period`, `documentNumber|null`, `documentDate|null`, `periodStart|null`, `periodEnd|null`, `region|null`, `managerLogin`, `managerName?`, `plan[]` (порожній `[]` якщо документа нема — **не помилка**).

**`ActivityPlanRow`:**
| Поле | Тип | Обов. | Опис |
|---|---|:---:|---|
| `category` | string | Так | RU-формулювання 1С: `Активный/Спящий/Потерянный/Без закупок/Новый`. |
| `planMeetings` | number | Так | Скільки **зустрічей** виконати по категорії. Головне (єдине зараз) поле. |
| `planCalls` | number | Ні | **Зарезервовано** (план дзвінків) — зараз **НЕ заповнювати** (опустити/`0`). Див. §10. |
| `totalInCategory` | number | Ні | Всього клієнтів категорії у менеджера (справково). |

**Логіка 1С:** знайти актуальний документ (`Менеджер.Логін==login`, період перетинається з місяцем, якщо кілька — останній за датою) → шапка + табличка у `plan[]`. Нема документа → `null`/`[]`, не помилка. Минулі/майбутні періоди — підтримувати (потрібна історія). Менеджера нема → `error`.
**Performance:** < 200мс, кешуємо 5хв.

### 7.2 `getActivityPlansBulk` — план по ВСІХ менеджерах (реалізувати РАЗОМ)

Головний сценарій — звід «план vs факт» по всіх менеджерах. Bulk за зразком `getRegistryPlans` (Action 4):
```json
{ "action": "getActivityPlansBulk", "payload": { "dateFrom": "2026-07-01", "dateTo": "2026-07-31" } }
```
```json
{ "status": "success", "data": { "plans": [
  { "managerLogin": "headofsd@emet.in.ua", "managerName": "Мігашко Анна", "region": "Київ", "category": "Активный", "planMeetings": 20 },
  { "managerLogin": "sm.kiev3@emet.in.ua", "managerName": "Селиванова Вікторія", "region": "Київ", "category": "Спящий", "planMeetings": 12 }
] } }
```
Плоский масив, рядок на (менеджер × категорія). Логіка та сама, без фільтра по login — усі актуальні документи, чий період перетинається з `[dateFrom;dateTo]`. Поля — як `ActivityPlanRow` + `managerLogin`/`managerName`/`region`. `planCalls` так само зарезервовано.

### 7.3 `getVisitsBulk` — вигрузка візитів (факт-історія)

Усі **виконані** візити по всіх менеджерах за діапазон — для наповнення історії + звірки.
```json
{ "action": "getVisitsBulk", "payload": { "dateFrom": "2026-01-01", "dateTo": "2026-01-31" } }
```
| Параметр | Тип | Обов. | Опис |
|---|---|:---:|---|
| `dateFrom`/`dateTo` | string | Так | Діапазон `YYYY-MM-DD` (по даті візиту), `dateTo` включно. |
| `login` | string | Ні | Якщо є — тільки цей менеджер (для точкової звірки). |

**Відповідь:**
```json
{ "status": "success", "data": { "visits": [
  { "docId": "0000001271320260604", "date": "2026-01-14",
    "managerLogin": "sm.kiev3@emet.in.ua", "managerName": "Селиванова Вікторія", "region": "Київ",
    "clientCode": "000008266", "clientName": "Beauty Generation ПК", "category": "Активный",
    "comment": "Презентували Neuronox, домовились на повторну зустріч", "status": "done" }
] } }
```
**`Visit`:**
| Поле | Тип | Обов. | Опис |
|---|---|:---:|---|
| `docId` | string | Так | Стабільний ID документа (не змінюється при редагуванні) — матчимо/оновлюємо по ньому. |
| `date` | string | Так | `YYYY-MM-DD` (можна з часом). |
| `managerLogin` | string | Так | Логін менеджера. |
| `managerName` | string | Ні | ПІБ. |
| `region` | string\|null | Так | Регіон/підрозділ (для розбивки). |
| `clientCode` | string | Так | Код контрагента (з провідними нулями). |
| `clientName` | string | Так | Назва. |
| `category` | string | Так | **Категорія на МІСЯЦЬ візиту** — з регістру категорій за той місяць (historical, не поточна). |
| `comment` | string\|null | Ні | Коментар менеджера. |
| `status` | string | Так | Повертати **тільки виконані** (`done`/«Проведено»). |

**Логіка 1С:** документи «Візит»/«Зустріч» зі статусом виконано, дата ∈ діапазон; (опц. фільтр login); категорія з регістру за місяць дати візиту; плоский `visits[]` (`[]` якщо нема — не помилка).
**Performance:** місячний зріз по всіх менеджерах — кілька секунд ок (тягнемо у фоні, чанкуємо по місяцях самі).

## 8. UI борду

- **Хедер-картки** (по компанії/регіону): всього план · виконано (унік.) · % · всього візитів. Стиль — `MetricCard`, як у weekly-report/дашбордах.
- **Розбивка регіон → менеджер** (акордеон): по кожному менеджеру таблиця категорій:

  | Категорія | План | Виконано (унік. клієнтів) | % | Всього візитів |
  |---|---|---|---|---|
  | Активні | 20 | 14 | 70% | 31 |
  | Сплячі | 10 | 6 | 60% | 8 |

- **Drill-down:** список контрагентів+дат візитів менеджера — рядок на візит: клієнт, категорія, дата, **коментар**. Згортається.
- Кольорове кодування % (good/warn/bad) — як на решті бордів. Переиспользуємо компоненти weekly-report (glass-cards, акордеон, drill-down).

## 9. Наша сторона — обсяг робіт

1. **`visits-sync.ts` + крон `/api/cron/sync-visits`** — копія `sales-sync` (backfill історії + помісячний долив + звірка; upsert по `docId`).

5. **Пункт меню** в account-dropdown (гейт як weekly-report).

Доступи — **без нової міграції** (переиспользуємо company-overview-permissions + resolveRegionOverrides).

## 10. Звонки — окремий майбутній пункт

- План звонків (`planCalls`) — **зарезервовано** в контракті `getActivityPlan`, зараз не заповнюється.
- **Факту звонків у нас НЕМАЄ** — лише boolean через `checkActivities` (Action 7), повноцінних подій дзвінків нема.
- Щоб рахувати план vs факт по **звонках** — потрібен окремий 1С-екшен «події дзвінків» (аналог `getVisitsBulk`, але для дзвінків). До того борд = **тільки зустрічі/візити**.
- Коли з'явиться: `planCalls` почнемо наповнювати (контракт готовий) + додамо колонки звонків у борд без ломки схеми.

## 11. Відкриті питання 1С-розробнику

1. **`getActivityPlan`:** точна назва документа у 1С («План активностей»?); набір категорій — рівно 5 (`Активный/Спящий/Потерянный/Без закупок/Новый`) чи інший?; структура таблички — по категоріях (як припускаємо)?
2. **`getVisitsBulk`:** чи може віддати **категорію на місяць візиту** з регістру категорій (для минулих місяців — historical)?; рік по всіх менеджерах — чанкувати по місяцях (ми й так чанкуємо)?
3. Клієнт без категорії в тому місяці → `category` = фактична з регістру або `"Без закупок"` за замовч. (узгодити).
