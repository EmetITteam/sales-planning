# Architecture Decision Records (ADR)

Реєстр архітектурних рішень для CRM-розширення sales-planning.
Кожен запис — point-in-time. Якщо рішення переглядається, додаємо новий запис зі статусом `Supersedes ADR-X` (не редагуємо старі).

**Контекст проекту:** Sales-planning перетворюється на повноцінну CRM EMET. Етапи:

- **Етап 0** (поточний): Архітектура + дизайн + 1С-спека
- **Етап 1:** Зустрічі (фіксація + геолокація — core)
- **Етап 2A:** Дебіторка (1С)
- **Етап 2B:** Рекламації (Bitrix24 + TG-бот, не 1С)
- **Етап 3:** Замовлення / реалізації

---

## ADR-1: 1С — source of truth, наш app — буфер + UI

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** Розширення у повноцінну CRM. Потенційний конфлікт «де живуть дані».

**Decision:** 1С залишається авторитетним джерелом всіх бізнес-даних (клієнти, замовлення, дебіторка, продажі). Sales-planning виступає UI-фронтом і буфером для роботи менеджера. Зустрічі — народжуються у менеджера, але кінцеве зберігання теж у 1С.

**Consequences:**
- Уся валідація бізнес-правил остаточно у 1С; ми робимо UX-валідацію наперед.
- Жоден модуль не має власної бізнес-логіки, що дублює 1С.
- Усе що записано локально — рано чи пізно йде у 1С.

## ADR-2: Buffer pattern для write-операцій

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** Швидкість UI важливіша за realtime sync. Менеджери не повинні чекати 1С на кожен клік. 1С може бути недоступна або повільна.

**Decision:** Write-операції (зустрічі, замовлення):

1. Миттєво записуються у наш Postgres з статусом `pending`.
2. Sync-worker (Vercel Cron або черга) раз на годину батчем відправляє у 1С з retry + DLQ.
3. UI показує статус `pending` / `syncing` / `synced` / `failed` (див. ADR-6).

Read-операції з кешуванням (TTL у ADR-8).

**Consequences:**
- Потрібна таблиця `<domain>_syncs` для audit trail.
- Потрібен sync-worker сервіс.
- UI обов'язково показує sync-статус, інакше менеджер не зрозуміє чи його дані дійшли.

## ADR-3: Refactor god-components — паралельно з фічами

**Date:** 2026-06-02 · **Status:** Accepted · **Supersedes:** TD-11/12/13 (P0 → integrated)

**Context:** Існують god-компоненти у sales-planning: `clients-page` 1855 рядків, `planning-form` 2272, `company-overview` 1176. У бэклозі це TD-11/12/13 «відкладено до v3». v3 = поточне розширення CRM.

**Decision:** Розбиваємо god-компоненти у момент торкання їх для нової фічі. Без окремої «фази рефакторингу». Заглядаючи у файл — виносимо найменш-зв'язану частину у окремий компонент. Цільова межа: ≤500 рядків на файл.

**Consequences:**
- Кожен спринт включає трохи рефакторингу.
- Не вкладаємо нові фічі у вже роздуті файли.
- Якщо дотик у god-component блокує спринт — створюємо TD-задачу і виносимо окремо.

## ADR-4: Увімкнути RLS перед першим sensitive-блоком

**Date:** 2026-06-02 · **Status:** Accepted · **Supersedes:** part of TD-4 (P3 backlog → P0)

**Context:** Зараз RLS вимкнено, скрізь service_role bypass (TD-4). Прийнятно для не-фінансових модулів. Дебіторка та замовлення — це гроші; рекламації — чутлива переписка.

**Decision:** Перш ніж писати першу таблицю з фінансово-чутливими даними (`debtors`, `orders`, `reclamations`) — увімкнути RLS на всіх існуючих + нових таблицях з per-row перевірками за `login`-claim з JWT. Закрити IDOR на `getClientReport` / `getAllMeetingsForClient`.

**Consequences:**
- Усі CRUD-роути перевіряють що користувач має право на конкретний клієнт/запис.
- JWT cookie несе login claim, який доступний у RLS-політиках.
- Раніше «магічних» service_role-обходів — не буде.

## ADR-5: Meeting-функціонал переписуємо на наш стек, не мігруємо код

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** `metting-4.0` = vanilla JS + Bootstrap 5, single-file SPA 2783 рядки. `sales-planning` = Next.js 16 + React 19 + TS + Tailwind. Прямий код-merge неможливий.

**Decision:** Беремо з meeting-4.0:

- UX-патерни і data model (як референс)
- 1С-action surface (контракт що зберігаємо)
- Текст інтерфейсу (legacy, менеджери звикли)

Не беремо:

- JS-код
- HTML-структуру
- CSS
- Бібліотеки (Bootstrap, LitePicker, iMask)

**Consequences:**
- Оцінка етапу 1 виростає (vs ілюзорний «лiфтнути готове»).
- Натомість отримуємо код у нашій стеці з тестами, типами і архітектурою.

## ADR-6: Sync-status — first-class UI-елемент

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** Buffer pattern (ADR-2) означає що дані можуть бути не-синхронізованими у момент перегляду. Без явного індикатора менеджер не розуміє чи його зустріч/заказ дійшли до 1С.

**Decision:** Кожен запис у CRM (зустріч, заказ, рекламація) має статус-бейдж у будь-якому view:

- `pending` — записано локально, чекає батч
- `syncing` — батч у процесі
- `synced` — підтверджено 1С (або Bitrix для рекламацій)
- `failed` — джерело відмовило, треба правка (показуємо причину)

Менеджер може редагувати запис у будь-якому статусі (немає інших менеджерів-конкурентів за конкретний запис — підтверджено користувачем).

**Consequences:**
- Реусабельний `<StatusBadge>` компонент.
- Sync-worker оновлює статус у `<domain>_syncs` таблицях.
- UI підписується через SWR з мутацією після батчу.

## ADR-7: Geolocation — read-only після захоплення, explicit на fail

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** У meeting-4.0 геолокація має п'ять болів: 10s timeout без конфігу, тихий null save, без retry, без manual fallback, lat/lon приховані.

**Decision:** У нашій реалізації:

- Geo-поля **read-only після захоплення** (менеджер не редагує адресу/lat/lon вручну для «правильно» зчитаної).
- Якщо захоплення впало (timeout / permission / signal) — **explicit toast** + кнопка **retry**.
- Опційно дозволяємо **ручне введення адреси** окремою кнопкою «Внести вручну» — lat/lon лишаються пустими, у БД явний прапорець `geo_manual=true`.
- Timeout конфігурований у env (`GEO_TIMEOUT_MS`, дефолт 15s).
- Lat/lon видно у формі (не приховані як у meeting-4.0).

**Consequences:**
- Менеджер у полі точно знає, що з геолокацією.
- Прихований null-save (поточний баг meeting-4.0) ліквідується.

## ADR-8: TTL для кешу read-операцій

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** Не дергаємо 1С на кожен клік (ADR-1, ADR-2). Read-дані кешуються через SWR з конкретним TTL.

**Decision:**

| Сутність | TTL | Чому |
|---|---|---|
| Дебіторка (взаєморозрахунки) | **10 хв** | Часто змінюється від касових операцій |
| Клієнти менеджера (список + базові поля) | **1 год** | Рідко змінюється |
| Підсумкові зустрічі (для дашборду) | **1 год** | Власне записані у нас → fresh, але refetch з 1С раз/год для звірки |
| Каталог продуктів (для заказів) | **6 год** | Майже не змінюється, важливий для UX |
| 1С action-меди (categories/regions/segments) | **24 год** | Майже статичні |

**Manual refresh кнопка** у кожному модулі — інвалідує SWR ключ.

**Consequences:**
- SWR конфігураційні константи у `src/lib/swr-config.ts`.
- Webhook-інвалідація з 1С — не зараз; повертаємось коли 1С підтримає.

## ADR-9: Failed-sync рядки повертаємо у UI як «потребує правки»

**Date:** 2026-06-02 · **Status:** Accepted

**Context:** Buffer-батч може містити 50 записів. 1С відхилила 1 рядок (наприклад, валідація). Решта 49 — приймаємо.

**Decision:**

- Sync-worker обробляє кожен рядок незалежно (per-record, не all-or-nothing).
- Failed-рядки залишаються у нашій БД зі статусом `failed` + `failure_reason` (текст від 1С).
- У UI: окремий зрозумілий індикатор «N записів потребує правки» зі швидким переходом на конкретний запис.
- Менеджер редагує — статус повертається у `pending` — наступний батч пробує знову.

**Consequences:**
- Sync-worker логіка retry потребує per-record state machine.
- Failed-рядки не блокують успішні.
- UI має зрозумілий "what's broken" індикатор у дашборді менеджера.

## ADR-19: Стратегічний KPI — % виконання з «Огляд компанії» (єдине джерело)

**Date:** 2026-07-03 · **Status:** Accepted

**Context:** Дашборд `/admin/strategic-kpi` спочатку рахував грошовий % виконання
власними викликами 1С (Action 4 план + Action 5 факт, `onec-plans.ts`). Виклики
company-wide важкі (30-40с), таймаутили в softRace → `onec_plans/onec_facts`
поверталися пусті → % відкочувався на **клієнтський** (Vitaran 90.9%, Ellanse
107.4% замість грошових). При цьому той самий % **вже коректно рахувався** в
«Огляд компанії» і в «Плануванні».

**Decision:**

- Дашборд НЕ робить власних 1С-викликів за планом/фактом. Фронт тягне готовий
  `/api/admin/company-overview` (хук `useCompanyOverviewExec`) і згортає у
  `brand × channel → {plan, fact}`. Джерело істини одне.
- % бренду = Σ факт / Σ план по каналах **зі стратег-тригерами** (`isChannelActive`):
  дистри → лише Ellanse, КЦ → лише ESSE/IUSE Coll./БАД, представництва → всі.
- Тільки місячний період (Огляд приймає `YYYY-MM`); квартал/рік → відкат на
  клієнтський %.
- `company-overview` тепер доступний і `isStrategicKpiLogin` (sdu@).

**Consequences:**

- Видалено `onec-plans.ts` та `segment_summary` (мертвий код).
- Розбіжність hero vs канал прибрана; на hero — розбивка по підрозділах коли
  каналів ≥2.
- Деталі: [`strategic-kpi-spec.md`](./strategic-kpi-spec.md).
