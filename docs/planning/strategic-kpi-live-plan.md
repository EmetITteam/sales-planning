# План доробок: Live-режим поточного місяця для дашборду «Стратегія»

**Дата:** 2026-07-03 · **Статус:** Draft (чекає «добро» на реалізацію)
**Залежність:** 1С action `getSalesLineItems` — [ONEC_STRATEGIC_LIVE_SPEC.md](../ONEC_STRATEGIC_LIVE_SPEC.md)

## Мета

Дашборд «Стратегія» має бути **постійно активним і актуальним**:
- **Минулі місяці** — з нашої БД (`sales`, вже заповнена, миттєво).
- **Поточний місяць** — свіжий, підтягується з 1С і докладається у БД.
- Без 30-хвилинних завантажень.

## Головна ідея оптимізації

**Дашборд ЗАВЖДИ читає з БД (миттєво), а свіжість поточного місяця забезпечує фоновий синк.** Користувач ніколи не чекає на 1С у момент відкриття.

Чому це швидко (обґрунтовано цифрами):
- Вся історія = 265 тис. рядків → **ніколи** не тягнеться з 1С повторно, лежить у БД.
- Один місяць = **~5–6 тис. рядків** (червень 5 522, травень 6 192) → легка вигрузка.
- Дашборд читає готові агрегати з БД (RPC + AsyncCache) — як зараз, миттєво.
- 1С викликається **у фоні** (cron), а не в request-path користувача.

```
┌─────────────┐   читає (миттєво, RPC)   ┌──────────────┐
│  Дашборд    │◀─────────────────────────│  Supabase    │
│ «Стратегія» │                          │  sales       │
└─────────────┘                          └──────▲───────┘
      │ «Оновити» (on-demand)                   │ upsert (mark-and-sweep)
      ▼                                         │
┌─────────────────────────────┐   fetch місяць  │
│ POST /api/analytics/        │────────────────▶│  1С getSalesLineItems
│ sales-sync                  │  (~5-6К рядків)  │  (тільки поточний місяць)
└─────────────────────────────┘
      ▲
      │ cron кожні 15 хв (робочий час)
```

Поточний місяць свіжий тому, що фон його докладає; дашборд його бачить одразу з БД. Це і є «real-time відстеження + дополнення базы».

## Архітектура синку

### Стратегія «full-month replace» (mark-and-sweep)

Кожен синк поточного (відкритого) місяця:
1. Тягнемо з 1С **весь поточний місяць** (пагінація по 1000).
2. Класифікуємо кожен рядок (бренд/канал/подарунок/ігнор/промо) — **тим самим класифікатором**, що backfill.
3. Upsert по `(doc_id, doc_line)` з `sync_id = <timestamp цього синку>`.
4. **Sweep:** `DELETE FROM sales WHERE sale_date у поточному місяці AND sync_id <> <цей sync_id> AND source='onec-live'` — прибирає рядки скасованих документів, яких уже нема у вигрузці.

Переваги: не треба «фід видалень» від 1С; місяць завжди консистентний; race-safe (спершу upsert, потім sweep — вікна «пусто» немає).

Закриті місяці — заморожені (синкнуті один раз після закриття, більше не чіпаються).

### Класифікатор — спільний модуль

Правила (BRAND_RULES / IGNORE / EXCLUDE / gift-detection / channel) зараз живуть у `scripts/analytics-sales-backfill.mjs`. Виносимо у `src/lib/strategic-kpi/sales-classifier.ts` (одне джерело істини) + юніт-тести. Backfill-скрипт і sync-endpoint імпортують з нього. Щоб правила не «розʼїхались».

| Фаза | Що робимо | Файли | Оцінка | Ризик |
|---|---|---|---|---|
| **0. 1С** | Реалізація `getSalesLineItems` (на боці 1С-розробника) | — | 1С-dev | залежність |
| **1. Класифікатор** | Портуємо правила у `src/lib/strategic-kpi/sales-classifier.ts` + юніт-тести; backfill-скрипт переводимо на нього | `sales-classifier.ts`, `analytics-sales-backfill.mjs`, `tests/` | 2–3 год | низький |
| **2. Sync-endpoint** | `POST /api/analytics/sales-sync` — тягне місяць з 1С (пагінація), класифікує, mark-and-sweep upsert, лог у `sales_import_batches`; захист від паралельних синків | `route.ts`, міграція (колонка `sync_id`/`source`) | 4–5 год | середній |

## Ендпоінт `POST /api/analytics/sales-sync`

- **Хто зве:** Vercel cron (авто) + кнопка «Оновити» (admin, on-demand).
- **Вхід:** `{ month?: "2026-07", force?: boolean }` — за замовчуванням поточний місяць.
- **Логіка:** `getSalesLineItems` (усі сторінки) → класифікація → mark-and-sweep upsert → запис у `sales_import_batches` (`source='onec-live'`, лічильники accepted/gift/ignored/excluded, `finished_at`).
- **Захист:** якщо вже є `in_progress` батч цього місяця < 5 хв тому — не запускаємо другий (проти паралельних cron+manual). `maxDuration` під розмір місяця.
- **Ідемпотентність:** повторний запуск нічого не ламає (upsert по ключу + sweep).

## Зміни у дашборді (read-path НЕ гальмуємо)

- Читання лишається з БД (як зараз) → миттєво.
- Додаємо у hero: **«Дані на HH:MM»** (час останнього успішного синку з `sales_import_batches`) + бейдж **LIVE** для поточного місяця.
- Кнопка **«Оновити»** → `POST /sales-sync?force` → після завершення re-fetch аналітики (інвалідація AsyncCache для поточного місяця).
- Минулі місяці — без бейджа LIVE (заморожені).

## Cron

- Vercel Cron у `vercel.json`: напр. кожні 15 хв 08:00–21:00 Київ (робочий час — коли реально йдуть продажі). Поза робочим часом — рідше/не треба.
- Cron зве `/api/analytics/sales-sync` (поточний місяць) + у grace-період ще й попередній.

## Оптимізація — чек-ліст

- ✅ Тягнемо тільки поточний місяць (~5–6К), не історію.
- ✅ Дашборд читає БД, не 1С (миттєво; 1С у фоні).
- ✅ Mark-and-sweep — без «фіда видалень», консистентно.
- ✅ AsyncCache інвалідуємо тільки для поточного місяця після синку.
- ✅ Пагінація 1000/стор. — маленькі відповіді.
- ✅ (Майбутнє) `changedSince`-дельта якщо місяць колись зросте.
- ✅ Індекс `idx_sales_sale_date` вже є (range-scan по місяцю).

## Ризики і мітигації

| Ризик | Мітигація |
|---|---|
| 1С повільний/впав під час синку | Синк фоновий → дашборд показує останні синкнуті дані. Cron повторить наступного тику. |
| Нестабільний `docLine` між синками → дублі | Просимо 1С давати стабільний `docLine`; sweep однаково прибере «осиротілі» рядки. |
| Розʼїзд правил класифікації (скрипт vs endpoint) | Спільний `sales-classifier.ts` + юніт-тести. |
| Паралельні cron+manual синки | Guard по `in_progress`-батчу + advisory-lock/idempotent upsert. |
| Ліміти Vercel cron (частота/тривалість) | 15 хв достатньо; `maxDuration` під розмір; пагінація не дає одному запиту роздутись. |
| Скасовані документи «залипають» | Full-month replace (sweep) прибирає їх кожен синк. |

## Відкриті питання (потрібне рішення користувача)

1. **Частота фонового синку:** 15 хв у робочий час — ок? Чи хочеш частіше (5 хв) / рідше (30 хв)? Компроміс свіжість vs навантаження на 1С.
2. **«Місяць тому» (попередній місяць):** беремо з БД (заморожений), але перші ~10 днів нового місяця ще синкаємо (пізні документи). Ок?
3. **Кнопка «Оновити»** доступна тільки admin/sdu, чи автосинку достатньо?

## Що НЕ входить (свідомо)

- Не міняємо джерело **% виконання** (лишається «Огляд компанії», ADR-19) — це план/факт, окремо від line-items.
- Не тягнемо історію повторно.
- Не переносимо класифікацію у 1С.
