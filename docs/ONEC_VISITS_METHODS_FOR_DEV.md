# 1С: три нових методи HTTP-сервісу для борду «Візити / Активності»

**Для:** 1С-розробник · **Дата:** 2026-07-16

Потрібно додати **3 нові дії** до існуючого web-сервісу (той самий endpoint, що `getRegionData` / `getClientReport` — просто новий `action` у payload). Авторизація як завжди: Basic Auth + `X-Service-Token`.

**Формат (стандартний):**
```json
// Запит
{ "action": "<actionName>", "payload": { ... } }
// Успіх
{ "status": "success", "data": { ... } }
// Помилка
{ "status": "error", "message": "..." }
```

Категорії клієнта скрізь у RU-формулюванні 1С: `Активный / Спящий / Потерянный / Без закупок / Новый`.

---

## Метод 1 — `getActivityPlan` (план активностей одного менеджера)

Повертає документ «План активностей» менеджера за місяць: скільки **зустрічей** треба виконати по кожній категорії клієнта. Це НЕ `getClientActivationPlan` (той про активацію клієнтів) — це про кількість зустрічей.

**Запит:**
```json
{ "action": "getActivityPlan", "payload": { "login": "headofsd@emet.in.ua", "period": "2026-07" } }
```
| Параметр | Тип | Обов. | Опис |
|---|---|:---:|---|
| `login` | string | Так | Логін менеджера. |
| `period` | string | Так | Місяць `YYYY-MM`. Документ, чий період перетинається з місяцем. |

**Відповідь:**
```json
{ "status": "success", "data": {
  "period": "2026-07",
  "documentNumber": "000000841",
  "documentDate": "2026-06-28",
  "periodStart": "2026-07-01",
  "periodEnd": "2026-07-31",
  "region": "Київ",
  "managerLogin": "headofsd@emet.in.ua",
  "managerName": "Мігашко Анна",
  "plan": [
    { "category": "Активный",    "planMeetings": 20, "totalInCategory": 45  },
    { "category": "Спящий",      "planMeetings": 10, "totalInCategory": 130 },
    { "category": "Потерянный",  "planMeetings": 5,  "totalInCategory": 200 },
    { "category": "Без закупок", "planMeetings": 8,  "totalInCategory": 150 },
    { "category": "Новый",       "planMeetings": 3,  "totalInCategory": 12  }
  ]
} }
```

**Поля шапки:** `period`, `documentNumber|null`, `documentDate|null`, `periodStart|null`, `periodEnd|null`, `region|null`, `managerLogin`, `managerName?`, `plan[]`.

**Рядок `plan[]`:**
| Поле | Тип | Обов. | Опис |
|---|---|:---:|---|
| `category` | string | Так | RU-категорія (5 значень вище). |
| `planMeetings` | number | Так | Скільки **зустрічей** виконати по категорії. Головне поле. |
| `planCalls` | number | Ні | **Зарезервовано на майбутнє (план дзвінків) — зараз НЕ заповнювати** (опустити або `0`). |
| `totalInCategory` | number | Ні | Всього клієнтів категорії у менеджера (справково). |

**Логіка:** знайти актуальний документ (`Менеджер.Логін == login`, період перетинається з місяцем; якщо кілька — останній за датою) → шапка + табличка у `plan[]`. Документа нема → `documentNumber: null`, `plan: []` (**не помилка**). Минулі й майбутні періоди підтримувати (потрібна історія). Менеджера нема у 1С → `status: error`.
**Швидкодія:** < 200мс.

## Метод 2 — `getActivityPlansBulk` (план по ВСІХ менеджерах)

Те саме, але одразу по всіх менеджерах за діапазон — головний сценарій зводу «план vs факт». Реалізувати **разом з методом 1** (та сама логіка, без фільтра по login). Аналог існуючого `getRegistryPlans`.

**Запит:**
```json
{ "action": "getActivityPlansBulk", "payload": { "dateFrom": "2026-07-01", "dateTo": "2026-07-31" } }
```

**Відповідь — плоский масив, рядок на (менеджер × категорія):**
```json
{ "status": "success", "data": { "plans": [
  { "managerLogin": "headofsd@emet.in.ua", "managerName": "Мігашко Анна",      "region": "Київ", "category": "Активный", "planMeetings": 20 },
  { "managerLogin": "sm.kiev3@emet.in.ua", "managerName": "Селиванова Вікторія", "region": "Київ", "category": "Спящий",   "planMeetings": 12 }
] } }
```
Поля рядка: `managerLogin`, `managerName`, `region`, `category`, `planMeetings` (+ `planCalls` так само зарезервовано, не заповнювати). Беруться всі актуальні документи, чий період перетинається з `[dateFrom; dateTo]`.

## Метод 3 — `getVisitsBulk` (факт візитів — вигрузка)

Усі **виконані** візити по всіх менеджерах за діапазон дат — для наповнення історії + звірки. (Поточні візити ведуться у нашому застосунку; цей метод потрібен для історії та контролю.)

**Запит:**
```json
{ "action": "getVisitsBulk", "payload": { "dateFrom": "2026-01-01", "dateTo": "2026-01-31" } }
```
| Параметр | Тип | Обов. | Опис |
|---|---|:---:|---|
| `dateFrom` / `dateTo` | string | Так | Діапазон `YYYY-MM-DD` по даті візиту, `dateTo` включно. |
| `login` | string | Ні | Якщо переданий — тільки цей менеджер (точкова звірка). |

**Відповідь:**
```json
{ "status": "success", "data": { "visits": [
  { "docId": "0000001271320260604",
    "date": "2026-01-14",
    "managerLogin": "sm.kiev3@emet.in.ua",
    "managerName": "Селиванова Вікторія",
    "region": "Київ",
    "clientCode": "000008266",
    "clientName": "Beauty Generation ПК",
    "category": "Активный",
    "comment": "Презентували Neuronox, домовились на повторну зустріч",
    "status": "done" }
] } }
```

**Рядок `visits[]`:**
| Поле | Тип | Обов. | Опис |
|---|---|:---:|---|
| `docId` | string | Так | Стабільний ID документа (НЕ змінюється при редагуванні) — по ньому оновлюємо/матчимо. |
| `date` | string | Так | `YYYY-MM-DD` (можна з часом). |
| `managerLogin` | string | Так | Логін менеджера. |
| `managerName` | string | Ні | ПІБ. |
| `region` | string\|null | Так | Регіон/підрозділ. |
| `clientCode` | string | Так | Код контрагента (з провідними нулями). |
| `clientName` | string | Так | Назва контрагента. |
| `category` | string | Так | Категорія клієнта **на МІСЯЦЬ візиту** (historical, з регістру категорій за той місяць — НЕ поточна). |
| `comment` | string\|null | Ні | Коментар менеджера до візиту. |
| `status` | string | Так | Повертати **тільки виконані** (`done` / «Проведено»). |

**Логіка:** документи «Візит»/«Зустріч» зі статусом виконано, дата ∈ діапазон; (опц. фільтр по `login`); категорія — з регістру категорій за місяць дати візиту; плоский `visits[]` (`[]` якщо нема — не помилка).
**Швидкодія:** місячний зріз по всіх менеджерах за кілька секунд — ок (тягнемо у фоні, ділимо по місяцях самі).

## Відкриті питання (прошу підтвердити перед реалізацією)

1. **`getActivityPlan`:** точна назва документа у 1С («План активностей»?); набір категорій — рівно 5 (`Активный / Спящий / Потерянный / Без закупок / Новый`) чи інший?; табличка справді по категоріях?
2. **`getVisitsBulk`:** чи можна віддати **категорію на місяць візиту** з регістру (для минулих місяців — historical)?; рік по всіх менеджерах — ок чанкувати по місяцях (ми й так так робимо)?
3. Клієнт без категорії у тому місяці → `category` = фактична з регістру чи `"Без закупок"` за замовч.?

**Дзвінки** (`planCalls`, факт дзвінків) — поки НЕ треба, зробимо окремим методом пізніше. Зараз тільки зустрічі/візити.
