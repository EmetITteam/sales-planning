# 07 · Пастки (gotchas)

## 🪤 Пастка 1: Дата-фільтр `<= end_date`

```sql
-- ❌ НЕПРАВИЛЬНО — пропустить весь день 25.05
WHERE sale_date <= '2026-05-25'

-- ✅ ПРАВИЛЬНО
WHERE sale_date < '2026-05-26'
-- або
WHERE sale_date < ('2026-05-25'::date + INTERVAL '1 day')
```

**Чому:** `<= '2026-05-25'` у Postgres означає `<= '2026-05-25 00:00:00'`. Усі записи з часом > 00:00:00 цього дня випадають.

**Доказ:** 2026-05-25 знайдено помилку — B2B 2026 рахувався 2 699 замість 2 705 (+6 клієнтів), B2C 1 110 замість 1 121 (+11).

**How to apply:** будь-який фільтр діапазону дат — переписувати на `<` з наступним днем. Особливо коли період закінчується "сьогодні" або фіксованою датою без часу.

## 🪤 Пастка 2: REF_DATE = today() (для recency)

```sql
-- ❌ НЕПРАВИЛЬНО
SELECT (NOW()::date - last_purchase) FROM ...

-- ✅ ПРАВИЛЬНО (REF_DATE = день після останньої дати в даних)
WITH ref AS (SELECT MAX(sale_date)::date + 1 AS d FROM sales)
SELECT (ref.d - last_purchase) FROM ... CROSS JOIN ref
```

**Чому:** дані з 1С можуть мати lag (sync вночі). Якщо REF_DATE = today() і даних ще немає за сьогодні — клієнти що купували вчора будуть мати recency = 1, а не 0. Дрібниця, але при backfill і повторах призводить до різних цифр на різних запусках.

## 🪤 Пастка 3: NBSP як thousand separator

**Тільки при парсингу 1С TSV — у Postgres вже не проблема** (дані зберігаються як numeric).

Згадка для backfill-скрипта:
```python
sum_p = sum_str.replace('\xa0', '').replace(' ', '').replace(',', '.')
```

## 🪤 Пастка 4: Перетин B2B і B2C

Сума колонок ≠ унікальних клієнтів. Якщо 2 клієнти купували і в B2B і в B2C — їх рахуємо один раз:

```sql
-- ❌ НЕПРАВИЛЬНО
SELECT
  (SELECT COUNT(DISTINCT client_code) FROM sales WHERE segment = 'B2B')
  + (SELECT COUNT(DISTINCT client_code) FROM sales WHERE segment = 'B2C')
  AS total;

-- ✅ ПРАВИЛЬНО
SELECT COUNT(DISTINCT client_code) FROM sales;
```

## 🪤 Пастка 5: "Сума колонок" у бакетах

Те саме що пастка 4, але для бакетів. Один клієнт за різні місяці може бути у різних бакетах. Тому:
- "Σ клієнто-місяців" — це сума колонок (валідно)
- "Σ унікальних клієнтів за період" — це `COUNT(DISTINCT client_code)` (НЕ сума колонок)

## 🪤 Пастка 6: Гіви/подарунки з sum > 0

Деякі промо ("1$ + Подарок 3 саше") мають sum_usd = 1, але це по суті безкоштовно. Без фільтра `sum_usd >= 5` вони фігурують як покупки.

## 🪤 Пастка 7: Семінар vs Проект (ELLANSE)

У вигрузці ELLANSE є поля `Семінар` і `Проект` — це РІЗНЕ. Подивись скрипти `ellanse_*.py` для деталей. Не плутати "купив на семінарі" з "проект клієнта".

## 🪤 Пастка 8: 1С INNER JOIN при додаванні поля

⚠️ **Зараз не актуально** (дані в Postgres), але якщо коли-небудь робиться окрема вигрузка з 1С:

При додаванні поля `Контрагент.Сотрудник` у звіт 1С робить INNER JOIN — клієнти з порожнім "Сотрудник" зникають. Якщо контрольна сума не сходиться зі старою вигрузкою — це найімовірніша причина. Просити вигрузку з LEFT JOIN.

## 🪤 Пастка 9: ESSE "Gift set 2026" не містить "esse"

При фільтрі бренду ESSE одне `WHERE product ILIKE '%esse%'` пропустить gift set. Завжди додавати `OR product ILIKE '%gift set 2026%'`.

## 🪤 Пастка 10: Округлення відсотків

При підрахунку `100 * a / b` в SQL з integer — отримаєш 0 для дрібних значень. Завжди:
```sql
ROUND(100.0 * a / NULLIF(b, 0), 1) AS pct
```
(і `NULLIF` щоб уникнути division by zero).

## 🪤 Пастка 11: Подвійні документи (Реализация + Возврат)

При запиті "скільки покупок" — рахувати тільки `Реализация`, не `Возврат`. У схемі є колонка `doc_id` яка має префікс — фільтрувати:
```sql
WHERE doc_id LIKE 'ЗИН%'
-- або через окрему колонку doc_type = 'Реализация'
```
