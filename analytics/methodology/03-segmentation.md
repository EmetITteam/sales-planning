# 03 · Сегментація B2B / B2C

## Базове правило

```sql
CASE
  WHEN division IN ('Коллцентр Call center лидогенерация', 'Интернет магазин esseskincare')
  THEN 'B2C'
  ELSE 'B2B'
END AS segment
```

У схемі це зберігається у колонці `segment` (вже обчислено при insert/sync).

## Хрест-пересічення

**За весь період 2025-01..2026-05:** лише **2 клієнти** купували і в B2B, і в B2C. Це винятки (вочевидь співробітники або помилка реєстрації). При підрахунку "усього унікальних = B2B ∪ B2C" не сума колонок, а UNION кодів.

```sql
-- Унікальних усього
SELECT COUNT(DISTINCT client_code) FROM sales WHERE ...;
-- НЕ так: SUM(b2b_count + b2c_count) ← переоцінка на 2
```

## Підрозділ vs Сегмент: коли важливо

- **B2B vs B2C** — стратегічна сегментація (як працюємо з клієнтом)
- **Підрозділ Київ vs Одеса** — операційна (хто менеджер, який ринок)

Для більшості запитів використовуємо `segment`. Якщо користувач запитує по регіону — фільтруємо `division`.

## Підрозділ як B2B/B2C — що робити з новими

Якщо у даних з'явилася нова `division` яка не у списку:
- За замовчуванням → B2B
- Перевірити з користувачем: чи це справді B2B регіон, чи новий B2C канал
- Оновити CASE-логіку у міграції

## How to apply
Якщо в запиті явно сказано "B2B" або "B2C" — фільтрувати по `segment`. Якщо сказано "Київ" або інший регіон — фільтрувати по `division`. Якщо не сказано — дати обидва розрізи.
