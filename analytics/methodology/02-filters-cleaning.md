# 02 · Фільтрація і очищення даних

⚠️ **Це найважливіший файл — без правильних exclusions цифри будуть зайво роздутими.**

## Стандартний "чистий" фільтр (золотий стандарт для більшості запитів)

```sql
SELECT *
FROM sales
WHERE sale_date >= '2025-01-01'                        -- період даних
  AND sum_usd >= 5                                     -- виключити саше за $1, тестери за $0.25
  AND discount IS DISTINCT FROM 'Рекламная продукция'  -- рекламна продукція не покупка
  AND product NOT ILIKE '%саше%'                       -- виключити саше за назвою
  AND product NOT ILIKE '%sachet%'
  AND client_code IS NOT NULL AND client_code != '';
```

## Що виключаємо і чому

### 1. `discount = 'Рекламная продукция'` (~12 800 рядків)
- Це безкоштовна роздача в рамках маркетингу
- Сума у цих рядках зазвичай $0, але іноді ставлять номінальну
- **Завжди виключати** з усіх підрахунків клієнтів і обсягів

### 2. `sum_usd <= 0` (повернення і подарунки)
- `sum < 0` — повернення товару
- `sum = 0` — подарунок у складі акції ("500$ Esse + Подарок Gift set 2026")
- **Завжди виключати** як "покупку"

### 3. `sum_usd < $5` — саше і тестери
- Промо "1$ + Подарок 3 саше" — клієнт платить $1 за саше
- Дешеві позиції $0.25-$0.35 — округлення/уцінка/тестери
- **Виключати** якщо рахуємо покупки реального товару. Залишити якщо рахуємо проникнення в саше-промо.

### 4. Саше / sachet у назві продукту
- Окремий критерій бо деякі саше можуть мати > $5 (зрідка)
- `product ILIKE '%саше%' OR product ILIKE '%sachet%'`
- **Завжди виключати**, якщо метою є реальні покупки

### 5. Шоти Collagen (`product ILIKE '%1 шот%'`)
- Подарунковий формат IUSE Marine Collagen — одна порція на промоушн
- **Для Collagen-аналізу:** залишити тільки `product ILIKE '%30 шотів%'` (повна упаковка)

### 6. Порожній `client_code`
- Іноді 1С пише сюди порожнечу (анонім або косяк введення)
- **Завжди виключати** з підрахунку клієнтів

## Особливі випадки

### Коли НЕ виключаємо подарунки і саше
- Аналіз ефективності промо-акції ("скільки клієнтів отримали Gift Set 2026")
- Шот-конверсія (хто отримав 1 шот → купив пачку)
- Cross-promo (купив ELLANSE з подарунком ESSE → потім купив ESSE окремо)

### Коли вмикаємо повернення
- Аналіз retention з урахуванням повернень
- Розрахунок чистого revenue (sum_usd може бути < 0)

## Параметри як константи

У playbook ми домовилися про дефолти:

| Параметр | Дефолт | Коли змінювати |
|---|---|---|
| `MIN_SUM` | $5 | для глибокого аналізу або по запиту |
| `EXCLUDE_SACHETS` | true | для аналізу саше-промо |
| `EXCLUDE_ADVERTISING` | true | майже завжди true |
| `INCLUDE_RETURNS` | false | при revenue-аналізі |

## Перевірка кількості виключених

Контрольний запит щоб бачити що скільки відсіює:

```sql
SELECT
  COUNT(*)                                                    AS total,
  COUNT(*) FILTER (WHERE discount = 'Рекламная продукция')     AS advertising,
  COUNT(*) FILTER (WHERE sum_usd <= 0)                         AS zero_or_negative,
  COUNT(*) FILTER (WHERE sum_usd > 0 AND sum_usd < 5)          AS very_cheap,
  COUNT(*) FILTER (WHERE product ILIKE '%саше%' OR product ILIKE '%sachet%') AS sachets,
  COUNT(*) FILTER (WHERE client_code IS NULL OR client_code = '') AS no_client
FROM sales
WHERE sale_date >= '2025-01-01';
```

## Why
Користувач (IT Director EMET) явно попросила перевіряти що ми не рахуємо саше і безкоштовні видачі як покупки. Раніше при підрахунку сплячих ESSE ми мали 191 → стало 190 (різниця -1) при додаванні цих фільтрів. Логіка важлива методологічно навіть якщо різниця мала.

## How to apply
Цей фільтр — дефолт для будь-якого запиту, де користувач не уточнив явно. Якщо в запиті є слова "промо", "акція", "подарунок", "саше" — спершу уточнити, чи виключати їх, чи рахувати окремо.
