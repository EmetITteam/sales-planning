# 01 · Джерела даних

## Архітектура

Раніше дані бралися з ручних TSV-вигрузок з 1С (звіт "Продажи"). У новому продукті дані живуть у **Supabase Postgres**, таблиця `sales`. 1С робить HTTP POST раз на день з інкрементом.

```
1С (УТП, Україна)
    ↓ HTTP POST /api/sales-sync (раз на день, інкремент)
Supabase Postgres
    ↓ SQL queries
AI-аналітик (через /api/analytics/chat)
```

## Таблиця `sales` (повна схема — див. [implementation/01-database-schema.md](../implementation/01-database-schema.md))

Скорочено — ключові колонки:

| Колонка | Тип | Опис |
|---|---|---|
| `doc_id` | text | Унікальний ID документа з 1С (ЗИН00034345) |
| `sale_date` | timestamptz | Дата і час реалізації |
| `client_code` | text | Код контрагента (9 цифр, з лідуючими нулями: `000017247`) |
| `client_name` | text | Назва клієнта |
| `phone` | text | Телефон |
| `product` | text | Повна назва номенклатури |
| `brand` | text | Бренд (виведений з product через regex, див. 05-brand-filters) |
| `discount` | text | Поле "Повод скидки" з 1С |
| `division` | text | Підрозділ (Київ, Одеса, Колл-центр, Інтернет-магазин esseskincare...) |
| `segment` | text | B2B / B2C (виведено з division) |
| `manager` | text | Менеджер (Сотрудник) |
| `qty` | numeric | Кількість в одиницях зберігання |
| `sum_usd` | numeric | Сума продажі без ПДВ у USD |
| `is_advertising` | boolean | `discount = 'Рекламная продукция'` |
| `is_sachet` | boolean | `product ILIKE '%саше%' OR '%sachet%'` |
| `is_gift_qty0` | boolean | `sum_usd = 0` |

## Звідки беруться

- **Період:** з 2025-01-01 і далі
- **Інкремент:** щоночі HTTP-сервіс шле новий день
- **Bulk backfill:** при першому запуску — імпорт історичної TSV-вигрузки (`data/База с 01.01.2025.txt`)

## Які значення `division` існують (станом на 2026-05)

### B2B (регіональні представництва)
- `Киев`, `Одесса`, `Днепр`, `Запорожье`, `Харьков`, `Николаев`, `Винница`, `Житомир`

### B2C
- `Коллцентр Call center лидогенерация` — це колл-центр (телефонні замовлення)
- `Интернет магазин esseskincare` — інтернет-магазин (флагман для ESSE)

> Список може поповнюватися. Логіка `segment = B2B/B2C` визначається в системі (SQL CASE або у view). Бачимо нову `division` яка не в списку — за замовчуванням B2B.

## Контрольні цифри (для перевірки коректності backfill)

Станом на період **2025-01-01 → 2026-05-25** з файлу `База с 01.01.2025.txt`:
- Усього валідних рядків (sum > 0): **79 715**
- Унік. клієнтів: **6 867** (B2B: 4 166, B2C: 2 703, перетин: 2)
- Контрольна сума: **$14 090 139.03** (відповідає рядку "Итог" у звіті 1С)
- Період даних: 2025-01-01 → 2026-05-25 (плюс 2 рядки-аномалії з 2024)

> Якщо після backfill контрольна сума не співпадає — щось загубилось при парсингу (найімовірніше NBSP або escape лапок).

## Як перевірити свіжість даних

```sql
SELECT MAX(sale_date)::date AS last_date,
       (CURRENT_DATE - MAX(sale_date)::date) AS days_behind,
       COUNT(*) AS total_rows
FROM sales;
```

Якщо `days_behind > 2` — HTTP-сервіс 1С зламався, треба перевірити.
